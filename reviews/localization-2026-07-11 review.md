## Evidence Inventory

### Message infrastructure

`pill-messages.ts` exports a full 17-locale CATALOG (`en`, `es`, `fr`, `de`, `it`, `pt`, `ru`, `el`, `zh-Hans`, `zh-Hant`, `ja`, `ko`, `hi`, `id`, `tr`, `vi`, `ar`, `he`) covering all keys of the `Messages` type except the optional `close?` field. The `resolveMessages` function auto-detects from `navigator.languages` with a `typeof navigator !== "undefined"` SSR guard, does BCP-47 matching via `pickKey` with zh-Hant/zh-Hans disambiguation, and falls back to `en`. `isRtlLocale` checks the primary subtag against `{"ar", "he", "fa", "ur"}`.

`compare-messages.ts` ships `defaultCompareMessages` — a single English-only catalog object. No per-locale entries, no `resolveMessages`-style resolution. The file's own comment states this is intentional: "M1a ships English content with the shared localization framework (fmt + isRtlLocale from pill-messages); the full per-locale catalog will be filled once the comparison surface stabilises through M1b/M1c." The scoping is documented, not accidental.

### Component sweep

**`EntvizPill.ts`:** All interactive chrome (menu labels, tooltip, aria-labels, rail steps, copy confirmations, close button) routes through `resolveMessages(locale)` → `m: Messages`. RTL is derived from the RESOLVED locale via `isRtlLocale(resolved)`, then overridden by the `dir` prop. The close button uses `m.close ?? "Close"` — the optional field falls back to an English literal at the call site (line 650).

**`EntvizCompare.ts`:** Chrome routes through `defaultCompareMessages` merged with host `messages` overrides (only English, scoped to M1a as documented). RTL detection on line 301: `const rtl = isRtlLocale(locale ?? "")`. When `locale` is `undefined` (the common case — no locale prop passed), this evaluates `isRtlLocale("")`, which returns `false` because `""` is not in the RTL set. As a result, an Arabic or Hebrew user who relies on auto-detection gets LTR chrome in the comparator. The `dir="rtl"` attribute on the root element is only set when this evaluates true.

**`EntvizWalk.ts`:** Entirely English. Copy lives in the module-level `M` object (not a Messages catalog). No `locale` prop, no `dir` prop, no RTL handling. Comment: "English walk copy (localization framework to follow once the surface settles)."

**`EntvizVoiceCompare.ts`:** Entirely English. Copy lives in the module-level `M` object. Comment: "English copy (the shared localization framework follows once the surface settles — same status as the M2 walk; full localization is issue #23)." No `locale`, no `dir`, no RTL.

**`Entviz.ts`:** The bare component renders no user-visible chrome (just the SVG). The `controls` toolbar (size ladder + reshape picker + copy/export kebab) contains hardcoded English: `aria-label="smaller"`, `aria-label="larger"`, `aria-label="size"`, `aria-label="shape"`, `aria-label="actions"`, and the `COPY_ACTIONS` / `COPY_TOAST` / `COPY_FAILED` constants (`"Copy value"`, `"Copy comparison text"`, `"Copy image"`, `"Copy SVG"`, `"Copied value"`, etc.). No `locale` prop, no message layer.

### Catalog key completeness

All 17 non-`en` CATALOG entries define the same 16 required `Messages` fields (all except the optional `close?`). No missing or empty values were found for required keys.

### RTL contract

The entviz SVG is generated by `@entviz/core`'s `render()` function, which produces a viewBox-anchored SVG with no direction-sensitive layout. It is direction-neutral. The pill's popover and menu carry `dir: dirAttr` (derived from the resolved locale + prop). `popCloseStyle` uses `insetInlineEnd: 6` (logical property — correct). The toast placement uses `[rtl ? "left" : "right"]: 0` (a physical-property workaround — functional but not logical-property clean, low severity).

`EntvizCompare`'s root element sets `dir: rtl ? "rtl" : undefined`, but `rtl` is broken under auto-detection as described above.

---

## Executive Summary

The `@entviz/react` chrome is **fully and correctly localized for `EntvizPill`**: 17 locales, complete key coverage, BCP-47 negotiation, SSR-safe auto-detection, and proper RTL chrome mirroring. The never-localize-the-viz contract is solidly honored — the entviz SVG is direction-neutral and never mirrored. The biggest gap is that **`EntvizCompare`, `EntvizWalk`, and `EntvizVoiceCompare` are English-only**, by documented milestone scoping — this is intentional but means the advertised 17-locale support does not yet extend to the comparison surface. Within `EntvizCompare` there is also a concrete bug: **RTL detection fails under auto-detected locales** (`locale` undefined → `isRtlLocale("") = false`), so Arabic/Hebrew users of the comparator see LTR layout even though the pill correctly mirrors. Additionally, `Entviz.ts`'s opt-in `controls` toolbar hardcodes English labels with no message layer hookup.

## Top Findings

### F1: `EntvizCompare` RTL detection silently broken under auto-detected locale

- **Severity:** HIGH | **Confidence:** CONFIRMED | **Location:** `packages/react/src/EntvizCompare.ts:301`
- **Finding:** `const rtl = isRtlLocale(locale ?? "")` — when `locale` is `undefined` (no explicit prop, the typical case), this calls `isRtlLocale("")`. The `isRtlLocale` function splits on `-` and checks the primary subtag against the RTL set `{"ar", "he", "fa", "ur"}`. `"".split("-")[0]` is `""`, which is never in the set, so `rtl` is always `false` when the locale is auto-detected. The comparator root element (`div` at line 794) sets `dir="rtl"` only when `rtl` is `true`. Result: an Arabic or Hebrew user (or any user whose `navigator.languages` resolves to `ar`/`he`) sees a LTR `EntvizCompare` layout regardless of their locale.
- **Consequence:** Arabic and Hebrew users of the comparison surface get the wrong visual direction. Layout flows LTR (flex row, text alignment, the acquisition textarea, the fetch button, provenance chips, verdict chip) instead of mirroring RTL. This breaks the RTL contract for a component that explicitly accepts a `locale` prop and passes it through to `EntvizVoiceCompare`.
- **Recommendation:** Use the same pattern as `EntvizPill`: call `resolveMessages(locale)` (or an equivalent that resolves the locale) and then pass the RESOLVED tag to `isRtlLocale`. One-line fix: `const { locale: resolved } = resolveMessages(locale); const rtl = isRtlLocale(resolved);`. This is low-effort and directly mirrors the working pattern in `EntvizPill`.
- **Fix effort:** small

### F2: `Entviz.ts` controls toolbar fully hardcoded English — no message layer

- **Severity:** MEDIUM | **Confidence:** CONFIRMED | **Location:** `packages/react/src/Entviz.ts:206–312`
- **Finding:** The opt-in `controls` toolbar rendered by `<Entviz controls>` (and thus exposed in the `EntvizPill`'s expanded Visualize state, the comparator figures, and the walk figures) contains entirely hardcoded English labels: `aria-label="smaller"`, `aria-label="larger"`, `aria-label="size"`, `aria-label="shape"`, `aria-label="actions"`, menu item labels `"Copy value" / "Copy comparison text" / "Copy image" / "Copy SVG"`, toast confirmations `"Copied value" / "Copied comparison text" / …`, and the error string `"Copy failed"`. None of these route through any message layer. `Entviz.ts` accepts no `locale` or `messages` prop.
- **Consequence:** A user in any of the 17 advertised locales who expands a pill and uses the controls toolbar sees English-only labels and confirmations — including the copy-action confirmations that the pill's own `Messages.copiedValue`/`copiedImage` etc. are designed to translate. The copy-confirmation labels in `Entviz.ts` duplicate the `Messages` keys in `pill-messages.ts` without consuming them. Under RTL, the controls toolbar is also direction-unaware (no `dir` attribute; no `insetInline*` on the toast `insetInlineEnd: 0`).
- **Recommendation:** Add `locale?: string` and `messages?: Partial<Messages>` props to `<Entviz>` and route the controls labels through `resolveMessages(locale)`. The `Messages` interface already defines the needed copy (`copyValue`, `copyComparison`, `copyImage`, `copySvg`, `copiedValue`, …). The size/shape/actions `aria-label` strings can be added as new keys or handled via a minimal internal Messages extension; the copy strings can be shared with `pill-messages.ts` directly. Fix also includes adding `dir` to the toolbar.
- **Fix effort:** medium

### F3: `close` key absent from all non-`en` CATALOG entries — English fallback always triggered

- **Severity:** MEDIUM | **Confidence:** CONFIRMED | **Location:** `packages/react/src/pill-messages.ts:31–79` + `EntvizPill.ts:650`
- **Finding:** The `Messages` interface marks `close` as optional (`close?: string`). The English catalog (`en`) defines it as `"Close"`. All 17 other CATALOG entries (es, fr, de, it, pt, ru, el, zh-Hans, zh-Hant, ja, ko, hi, id, tr, vi, ar, he) omit the field entirely. At the call site in `EntvizPill.ts:650`: `"aria-label": m.close ?? "Close"` — the fallback is the English literal. Since no non-`en` catalog entry has `close`, this means the close-button aria-label (and its `title` tooltip) is always `"Close"` in English for every non-English locale. The field's own JSDoc comment acknowledges this: "Optional — falls back to English 'Close' when a locale bundle omits it (an icon label; native review pending)."
- **Consequence:** Screen-reader users in Arabic, Hebrew, French, or any of the 16 non-English locales hear the English label "Close" on the popover's ✕ button. The tooltip (shown on hover) also shows English. This is an accessibility gap baked into the localization coverage.
- **Recommendation:** Populate `close` in all 17 CATALOG entries. Machine-drafted translations are consistent with the rest of the catalog quality (already caveatted as "demo quality / not native-reviewed"). The `??` fallback approach is safe after the field is populated — it handles older/host-supplied partial `messages` bundles that predate the key. Remove the "native review pending" caveat once populated (same status as the rest of the catalog).
- **Fix effort:** small

### F4: `EntvizWalk` panel labels "Yours" and "Reference" are hardcoded English

- **Severity:** MEDIUM | **Confidence:** CONFIRMED | **Location:** `packages/react/src/EntvizWalk.ts:381–382`
- **Finding:** The `panel()` helper renders `<span>` labels above each figure. The calls at lines 381–382 pass `"Yours"` and `"Reference"` as string literals. These strings are also in `defaultCompareMessages.yours` and `defaultCompareMessages.reference` (the compare messages catalog), but `EntvizWalk` does not import or use `CompareMessages` — it has no `messages` prop and no message layer at all. All 35+ strings in `EntvizWalk`'s `M` object are hardcoded English.
- **Consequence:** An Arabic or French user walking the cells sees "Yours" / "Reference" as panel labels even if they are using `EntvizCompare` (which has its own `m.yours` / `m.reference` in `defaultCompareMessages`). The walk's other copy (button labels, coverage meter ticks, verdict messages, prompts for each feature kind, probe notice) is also English-only. This is the same deferred-localization category as `EntvizVoiceCompare`.
- **Recommendation:** Same milestone-sequenced approach noted in the code and issue #23: once the walk surface stabilises (M2b), add `locale?: string` and `messages?: Partial<WalkMessages>` props and route `M` through a message layer. For the near term, at minimum extract `"Yours"` and `"Reference"` to the `CompareMessages` keys already defined (they can be passed in from `EntvizCompare` which already has `m.yours` / `m.reference`), since `EntvizWalk` is invoked from `EntvizCompare` which has the locale context.
- **Fix effort:** medium (full), small (just Yours/Reference from parent)

### F5: `EntvizVoiceCompare` entirely English-only including security-critical prompts

- **Severity:** MEDIUM | **Confidence:** CONFIRMED | **Location:** `packages/react/src/EntvizVoiceCompare.ts:79–109`
- **Finding:** All copy in `EntvizVoiceCompare` is hardcoded English in the `M` object. This includes the §15.1 affirmation prompt ("Comparing by voice checks one thing: whether the person on your call is looking at the same value as you. It can't tell you who they are…"), the per-cell instruction ("Have the other party read ${address} aloud. Does it match what you see?"), the final verdict strings, the recognition-note scoping caveat, and all button labels. No `locale`, `messages`, or `dir` prop. Issue #23 is called out in a comment.
- **Consequence:** The voice ceremony is the only genuinely two-party interaction path in the system. Its §15.1 affirmation prompt is a security expectation-setter that a non-English-speaking user must understand to use the ceremony safely. An Arabic or Hindi user reading an English prompt may not fully grasp the limitations being stated ("it can only rule out a man-in-the-middle if your voice channel itself has integrity"). The hardcoded instruction "Have the other party read ${address} aloud" is also inaccessible to non-English speakers. This is a more consequential English-only surface than a tooltip or button label.
- **Recommendation:** Prioritize localization of the affirmation prompt and the §15.1 conditional clause in the verdict strings for the M3 milestone (when the voice ceremony ships to end users). The per-cell instruction is a natural-language template that should use `fmt`-style interpolation once the message layer is added. Add `locale` and `messages` props following the `EntvizCompare` pattern.
- **Fix effort:** medium

## Additional Gaps Noted

- **`EntvizCompare` locale prop is not forwarded to `EntvizVoiceCompare`:** `EntvizCompare` calls `h(EntvizVoiceCompare, { value, ..., rng })` (line 771) but omits `locale`. When the voice tab is offered, the locale context is lost. Low impact while `EntvizVoiceCompare` is English-only; will matter once it's localized.
- **`EntvizCompare` locale prop is not forwarded to `EntvizWalk`:** Similarly, `h(EntvizWalk, { ..., rng })` at line 744 omits `locale`. Same status.
- **Toast placement physical property (`EntvizPill.ts:676`):** `[rtl ? "left" : "right"]: 0` uses a physical CSS property rather than `insetInlineEnd: 0`. Functional under RTL but not consistent with the logical-property approach used in `popCloseStyle`. Very low severity.
- **`isRtlLocale` RTL set incomplete:** `{"ar", "he", "fa", "ur"}` is in the set, but `fa` (Farsi/Persian) and `ur` (Urdu) do not have CATALOG entries. If a user has `fa` or `ur` in `navigator.languages`, `isRtlLocale` correctly identifies them as RTL, but `resolveMessages` falls back to English (no catalog entry). This is consistent behavior but means RTL chrome is applied without a translated catalog — a user gets correct RTL direction but English text. Minor, since `fa`/`ur` are not in the target locale set.
- **`entviz visualization` accessible label in `Entviz.ts:137` is English-only:** `const defaultLabel = note ? \`entviz visualization, note ${note}\` : "entviz visualization"`. This string is the `role="img"` aria-label for the expanded entviz, used by screen readers to announce what it is. It is not translated. `"entviz"` is a proper noun and reasonable to keep as-is, but the surrounding wrapper text "visualization, note" is English. Severity is low because the word "entviz" itself is the primary identity marker, and the note (if any) is user-supplied.

## What's Done Well

1. **The message infrastructure in `pill-messages.ts` is well-designed.** `resolveMessages` is SSR-safe, handles BCP-47 region subtags, has a correct zh-Hant/zh-Hans split, falls back gracefully to `en`, and exposes `SUPPORTED_LOCALES`. The `fmt` function is simple and correct for the template patterns used.

2. **`EntvizPill` is fully and correctly localized.** Every user-visible label — tooltip, aria-labels, copy menu, copy confirmations, rail steps, teaching header, compare action — routes through the `m: Messages` bundle. RTL is properly derived from the resolved locale and overridable via the `dir` prop. Logical CSS properties (`paddingInlineStart`, `paddingInlineEnd`, `insetInlineEnd`) are used for RTL-sensitive layout.

3. **The never-localize-the-viz contract is solidly honored.** The `render()` call produces direction-neutral SVG. No locale-aware case operations touch the `value` or comparison text. Verdict strings in `EntvizCompare` are locked via `VERDICT_LOCKED_KEYS` so host locale overrides cannot relabel a verdict.

4. **The `close?` optional-field design is smart.** Making it optional with a fallback means a host can supply a partial `messages` bundle without breaking the close button — the fallback behavior degrades gracefully even if not ideally.

5. **Catalog coverage is complete for the supported locales.** All 17 CATALOG entries define all required `Messages` keys. No silent missing-key gaps that would leave a translated UI partially English.

## Residual Unknowns

- **Visual RTL layout correctness in `EntvizPill`:** The RTL floater placement math in `placeFloater` (right-aligns when `rtl=true`) and the `dir: dirAttr` attribute are correct in code. Confirming that the popover, menu, and toast visually mirror correctly (badge moves to the right, menu opens on the left, etc.) requires a running RTL build. Test: render `<EntvizPill locale="ar" value="..." />` and confirm layout mirrors.
- **`EntvizCompare` RTL after the fix:** Once `isRtlLocale(resolved)` is used, the layout direction change needs visual verification — the flex row of figures, the acquisition textarea + fetch button, the verdict chip alignment, the provenance label should all flip. Test: render `<EntvizCompare locale="ar" value="..." />` and verify.
- **`close` translations quality:** Machine-drafted translations of "Close" into 17 languages are simple (most are one word) but should have native review for RTL locales especially, where the button title tooltip direction context matters.
