# Red-Team Report — Lens: Rendering, Image Forensics & Artifact Spoofing

**Reviewer mandate:** break the comparison feature; weight everything toward any path to a false "verified"/"identical." I lead with rendering/forensics/artifact-spoofing and range into adjacent domains where the thread bites. My single most important conclusion up front: **the design's central soundness claim — "text is unmatchable, so it anchors the meter" — is a claim about *characters*, but every machine engine and every human read in this feature operates on *glyphs and pixels*, and the spec itself hands an attacker the tools to drive those two apart.** That gap is where most of my findings live.

---

## FINDINGS

### F1 — CRITICAL: The "text-anchored soundness" claim is defined over characters, but the SVG engine (C-SVG) verifies the value by *trusting `<text>` element content / data-attributes*, which can disagree with the displayed glyphs.

**Attack/failure scenario (attacker tier: T1 + T2, hand-authored SVG).**
Decision C-SVG says: for the SVG engine, "reconstruct normalized core from cell texts ≤512" and compare *at the value level*; decision D/F say soundness is anchored in the text channel because "an attacker can't construct a real key whose tokens equal the target's." But the SVG engine recovers cell text from the SVG's `<text>` nodes (Tier-A render-model recovery; spec *SVG profile*, *The render model (Tier A)*). The Tier-B raster comparison — the only thing that proves *displayed pixels* match the render model — **explicitly excludes text-glyph regions** ("Text-glyph regions are excluded from the pixel comparison… the visible glyph depends on which monospace font the environment provides," spec *Canonical rasterization (Tier B)*).

So I hand the tool an SVG where:
- the `<text>` node for cell 7 contains the characters of the *target's* token (so value-level reconstruction reproduces the reference core and the engine concludes "NO DIFFERENCE / IDENTICAL");
- but the cell visually shows something else, because I attach `dx`/`dy` tspan kerning, a sub-pixel `transform`, a `fill="#nucleusbg"` (text painted in the nucleus's own background color = invisible) on the real character plus a second decoy `<text>`, or simply rely on the renderer's font to draw a confusable glyph (F3).

The closed-profile checker (spec *Closed profile*) constrains *element types and paint order*, not whether `<text>` content equals displayed ink — and it cannot, because Tier B was *designed* to ignore glyph pixels. **The feature's own value-level SVG engine therefore certifies sameness from bytes that the human is told to trust as "the same picture," while the picture can show different glyphs.** This is a direct route to machine-**IDENTICAL** (decision A reserves IDENTICAL for "pasted SVG… machine compared both sides in full") on two values whose rendered text differs.

**Document grounding.** Spec *Canonical rasterization (Tier B)* (text-region exclusion); spec *The render model (Tier A)* and *SVG profile* (value recovered from attributes/`<text>`, not pixels); brief decision C-SVG, D, F; threat-model T2 ("control the rendering surface… screenshot vs. live render"). The paper §2.3 is the theoretical root: "A channel is only as discrete as the viewer's willingness to decode it one element at a time" — here the *machine* decodes the character stream and never checks it against ink.

**Recommended fix.** The SVG engine MUST NOT trust `<text>` content or `data-*` as the source of truth for an attacker-supplied reference. Either (a) re-render the reference SVG's cell text through the tool's *own* pinned font and compare the resulting glyph rasters to ours within the cell bounding boxes (i.e., **do not** inherit Tier B's text-exclusion for the comparison feature — that exclusion exists for cross-platform conformance, not adversarial comparison), or (b) downgrade any SVG whose `<text>` nodes carry positioning overrides (`dx`/`dy`/`transform`/multiple tspans/zero-contrast fill) to UNKNOWN and force the guided human walk. A pasted-SVG path that reaches IDENTICAL while ignoring glyph pixels is unsound.

### F2 — CRITICAL: Auto-detect (decision B) routes a polyglot SVG to the most permissive engine; an SVG that *rasterizes differently than its render model* defeats both the SVG engine and the human.

**Attack scenario (T1+T2).** Decision B "auto-detects text vs SVG vs raster vs URL," and decision C ranks the engines by permissiveness: SVG can reach IDENTICAL, raster *cannot prove equal* (DIFFERENT/UNKNOWN only). The lens question — "does auto-detect ever route an attacker artifact to a more permissive engine?" — has a clean yes.

I author a file that is a **valid SVG** but whose render diverges from its declared model:
- **foreignObject / CSS injection.** The spec's closed profile forbids `foreignObject`, but the *comparison feature's* auto-detector is not the conformance checker — it must decide "is this an SVG?" before deciding "is it conformant?" If it parses cell text from the DOM before running `validate_closed_profile`, a `<foreignObject>` or a `<style>` block with `text { fill: transparent }` / `@font-face` substituting a homoglyph font alters the *displayed* entviz while the `<text>` nodes (which the value engine reads) say the target's characters. If it *does* run the closed-profile check first and rejects, then the attacker downgrades: ship the same bytes so they're sniffed as a **raster** (see F6 polyglot) — but raster can only DIFFERENT/UNKNOWN, which is the *safe* direction, so the attacker instead wants the SVG path. The danger is precisely that SVG is the only auto-path that yields IDENTICAL.
- **External-resource / late-binding render.** `<image href="https://attacker/x.png">`, `<use href="...">`, or a CSS `url()` font means the bytes the engine hashes/parses are *not* the pixels the human later sees (the resource loads after, or differs by `prefers-color-scheme`, viewport, or `@media print`). The engine reconstructs a clean value from the static `<text>`; the human sees attacker-controlled late-bound content.

**Document grounding.** Brief decision B (auto-detect), C (per-medium permissiveness, SVG→IDENTICAL, raster→never-EQUAL); spec *Closed profile* (which the comparison engine is **not** identical to and may run in the wrong order); threat-model T2; secondary win "Inject script or stylable content via the rendered SVG that affects the embedding page." The asymmetry that raster is fail-safe but SVG is not is the exploitable seam.

**Recommended fix.** Auto-detect MUST run a strict closed-profile + structural validator *before* extracting any value, and MUST reject (route to human walk, never IDENTICAL) any reference SVG containing `<foreignObject>`, `<style>`, `<image>`, external `href`/`url()`, `@font-face`, `transform` on text, or media queries. Treat "is this conformant and self-contained?" as a gate, not an afterthought. An SVG that references anything external is by definition not a value the machine can compare "in full," so it must not reach IDENTICAL.

### F3 — CRITICAL: Homoglyphs across the spec's font-fallback chain break the "unmatchable text" claim at the visual layer — for the *human* read in the live/raster-fallback walk (decision D, E).

**Attack scenario (T1+T4, live mode or guided walk).** The text channel is the security anchor (decision D, paper §5.3–5.4). But the spec's own *Font-family fallback chain and homoglyph risk* section is candid: "characters like `0`/`O`, `1`/`I`/`l`/`|`, `5`/`S`, and `-`/`_` are visually-distinguishable in some monospace fonts and visually-confusable in others… an entviz is only as trustworthy as the user's ability to distinguish characters." The chain `"JetBrains Mono", "Menlo", "Consolas", "DejaVu Sans Mono", … monospace` routes different viewers to different fonts, and the spec accepts that glyphs are *not* identical across platforms (Tier B excludes them for exactly this reason).

In the **live two-party walk** (decision E) the counterparty reads a value aloud and *my* viewer reads mine. The soundness argument (decision D/F) says cell text is "unmatchable" because an attacker can't make a *real key* whose tokens equal the target's in chosen positions. But that argument is about *characters*, and the human is comparing *glyphs*. I (a T1 attacker who chose one value) don't need character equality — I need the target's value and mine to *render to confusable glyphs* in a position the walk happens to check:

- Target token `l0O1` vs. my token `I0O1` (or `1OO1`, `lOO1`…). On a viewer whose fallback resolved to a low-disambiguation monospace (the bare-`monospace` worst case the spec warns against, or Consolas where `l`/`I` are close), these are visually identical. The careful-reader assumption ("read it aloud, taking into account case") collapses because the cap/dash/under convention (spec *Thoughts About Comparing*) helps *case* and `-`/`_`, but does nothing for `0`/`O` or `1`/`l`/`I` — those have no spoken disambiguator in the convention.
- I am NOT trying to forge a full key; I am trying to make *one or two* of the front-loaded text checks (decision D front-loads text) read as a match on the specific viewer's font. Because the walk credits **viewer-relative bits** (decision D, paper §5.4) and reaches "verified" *primarily via text*, a handful of homoglyph-passing text cells is exactly the budget the attacker needs to exhaust.

This is the deepest finding against the design's thesis: **the "unmatchable text" claim is true over the character alphabet and false over the rendered glyph set, and every consumer of the claim (the meter, the walk, the human) operates on glyphs.** The spec admits the glyph gap exists; the comparison feature then leans its entire soundness story on the channel that has it.

**Document grounding.** Spec *Font-family fallback chain and homoglyph risk* (explicit homoglyph admission, platform-dependent fonts, Tier-B text exclusion); spec *Thoughts About Comparing* (cap/dash/under convention covers case and `-`/`_` only); paper §5.3 ("read every cell aloud, with case, has verified every bit" — assumes glyph=character), §5.4 (viewer-relative bits, text as the color-independent anchor); brief decision D ("for high-entropy values cell text is unmatchable"), human-factors pool item (`0`/`O`, `1`/`l`/`I`, `5`/`S`). Threat-model T4 ("substitute a hostile or narrow-glyph font").

**Recommended fix.** (1) The walk MUST credit a text cell only after disambiguating homoglyph characters with an explicit spoken/visual protocol (e.g., NATO-style readout for `0`="zero" vs `O`="cap-oh", `1`="one" vs `l`="ell" vs `I`="cap-eye"), not the bare cap/dash/under convention. (2) The tool MUST render comparison text in an *embedded, pinned* high-disambiguation font (the spec's "out of scope" font-embedding is in scope for a *verification* tool, even if not for the conformant artifact), so the security-critical read does not depend on the viewer's accidental fallback. (3) Down-weight or refuse to credit any text cell containing a homoglyph-class character unless rendered in the pinned font. Without (1)–(3), the "text-anchored meter" over-credits and decisions D and F do **not** survive (see Lens Verdict).

### F4 — HIGH: The raster fidelity self-probe (decision C) is defeated by a partial-fidelity image — exact probe regions, hand-tuned everywhere else.

**Attack scenario (T1+T2, hand-authored raster).** Decision C: the raster engine probes "known-exact regions (bounding fill `#ffffff`, borders `#808080`, color-bar bands = exact palette entries); if exact, trust nucleus/colorbar/ellipse sampling enough to **disprove** on mismatch." The inference is "the whole image is high-fidelity because these regions are." That inference is false for an attacker-authored raster, because **the probe regions and the misleading regions are spatially disjoint and independently controllable in a hand-built PNG.**

Construction (concrete bytes):
1. Start from a true entviz raster of the *target* (right colors/layout). 
2. Set every probed pixel exactly: bounding-rect fill literally `0xFFFFFF`, the 1-px `#808080` borders (spec: drawn with `shape-rendering="crispEdges"`, so they're axis-aligned exact pixel rows/columns — trivial to reproduce exactly), and the color-bar band fills as exact palette hex (`#ffffff/#e7be00/#ff3f2f/#2f3fbf/#000000`). Use a **lossless PNG, no ICC profile, no chroma subsampling** so these come back bit-exact. The probe passes ("trustworthy enough to disprove").
3. But the *text* — the only identity channel, which the engine does **not** OCR (decision C, explicitly) — is the *attacker's* value, hand-painted. Nucleus colors are entropy-derived; since color and text "are not bound in an attacker-authored image" (decision C's own premise), I paint the *target's* nucleus colors over *my* text. Now: probe passes → engine trusts nucleus/colorbar/ellipse sampling → nucleus/colorbar/ellipse all **match the target** (I copied them) → engine finds **no mismatch to disprove on** → emits **UNKNOWN/NO-DIFFERENCE**, never DIFFERENT.

The raster engine "can only disprove." I've removed every disprovable signal (because I copied all the sampled channels from the target) while the *text* (un-sampled, un-OCR'd) is wrong. The engine cannot reach EQUAL — correct — but it *also* cannot reach DIFFERENT, so it outputs UNKNOWN and **hands a guided human walk** a raster in which every gestalt channel matches and only the un-highlighted text differs. Combined with F3 (homoglyph text) and F8 (overlay/walk misalignment), the human then fails to catch the text difference. The fidelity probe gave *false assurance to the next stage*, which is the actual harm: it certified "this raster is trustworthy enough to sample," and a downstream walk inherits that trust.

A subtler variant defeats the probe's "if degraded, bail" branch in the *opposite* direction: make the probe regions *slightly* off (e.g., `#fffffe` from a deliberately near-lossless JPEG at quality 98 with 4:4:4 subsampling) so the engine bails to UNKNOWN/human — a **denial / forced-fallback lever** (F7).

**Document grounding.** Brief decision C (raster engine, fidelity self-probe, no OCR, "color and text are not bound in an attacker-authored image," "may only output DIFFERENT or UNKNOWN"); spec *bounding rect* (`crispEdges`, exact `#808080`/`#ffffff`), color-bar band exact palette; paper §3.2 (perceptual-hash-as-EQUAL is the attacker's goal — here the engine avoids EQUAL but the *probe's trust inference* is the perceptual-hash mistake one layer up).

**Recommended fix.** The probe must not infer whole-image fidelity from disjoint exact regions. Either (a) require the *text regions* themselves to pass a fidelity check (which forces OCR or glyph-template matching — i.e., abandon "we do not OCR" for raster references that reach any positive verdict), or (b) make the raster engine's positive output strictly DIFFERENT-or-nothing with **no UNKNOWN that carries forward trust** — UNKNOWN must reset the human walk to *zero credited bits*, treating the raster as if absent, never as "gestalt already confirmed." State explicitly that a passed fidelity probe credits **no** comparison bits; it only licenses *disproof*.

### F5 — HIGH: Chroma subsampling / dithering at nucleus edges and the off-center sample point lets a true mismatch survive the raster disprove-check.

**Attack scenario (T2, lossy-pipeline raster — even a non-malicious screenshot).** The raster engine samples nucleus/colorbar/ellipse to disprove on mismatch. Nucleus fills are 24-bit RGB (16M colors), and the design itself says fine gradations "may not be perceptible" and "will disappear if less than 16 million colors are displayable" (spec *Guarantees*, nucleus channel). A reference raster that went through **4:2:0 chroma subsampling** (default JPEG, default screen-capture-to-JPEG, many screen-share codecs in *live* mode) averages chroma over 2×2 blocks. At a nucleus edge — where the saturated nucleus meets the `#ffffff` margin or a surround box — the sampled chroma is a *blend*, so two nuclei differing by a few low-order RGB bits (precisely the bits the design admits are sub-JND) sample to the *same* subsampled value. The disprove check needs an *exact* mismatch to fire DIFFERENT; subsampling and dithering erase exactly the small mismatches, so a true small-difference value **survives** the raster auto-check as NO-DIFFERENCE/UNKNOWN.

This matters most in **live mode over screen-share** (decision E context, "another person on a live call"): the video codec is a lossy raster pipeline the design does not control, and the nucleus channel — already only a "hint" — is destroyed as a disproof channel, throwing the entire burden onto text (F3) and the discrete landmarks.

**Document grounding.** Spec *Guarantees* (nucleus is a non-primary hint, sub-JND gradations vanish under reduced color depth); threat-model accepted risk "Fine nucleus color gradations under reduced color depth"; brief decision C (sample nucleus to disprove), E (live call). Lens emphasis: "Chroma subsampling, dithering… scaling/AA at nucleus edges and the off-center sample point."

**Recommended fix.** The raster engine must detect a lossy/subsampled pipeline (probe for non-4:4:4 chroma, JPEG quantization artifacts, ICC presence) and, when found, **exclude nucleus color from disproof entirely** (it is unreliable, per the spec's own caveat) and refuse to credit it — relying only on the discrete, high-contrast channels (border presence, color-bar band *letters* if legible, blank-map shape positions). Never let "nuclei sampled equal" count as evidence of sameness under a lossy pipeline.

### F6 — HIGH: SVG/raster polyglot defeats medium auto-detection (decision B), and the closed-profile/AR-reconstruction boundary can be straddled.

**Attack scenario (T1+T2).** A single file can be sniffed as **both** SVG and raster depending on the detector's heuristic order. A `.svg` whose leading bytes also satisfy a raster magic, or an HTML/data-URL drop where the MIME and the content disagree, lets the attacker *choose* which engine fires. Because SVG is the only engine that yields IDENTICAL and raster is fail-safe, the attacker steers to SVG; because the SVG value-engine trusts `<text>` (F1), the polyglot's *visual* layer can be the raster (showing the attacker's value) while its *XML* layer carries the target's cell text. The "auto-detect" convenience of decision B removes the user's ability to assert "treat this as a raster (untrusted) only."

A second straddle exploits **C-SVG's AR-invariant reconstruction**: for >512-bit references the SVG engine compares "AR-invariant fingerprint-derived fields." Those fields (surround bits via `data-surround-bits`, color-bar `data-*`, blank-map `data-blank-map-*`) are *declared attributes a checker reads directly* (spec *SVG profile*: "A checker MUST recover the surround bits and edge color from these attributes, **not** by measuring box geometry"). So a hand-authored SVG can set `data-surround-bits`, `data-blank-map-min/max`, `data-color-bar-*` to the **target's** fingerprint-derived values while the actually-painted boxes/markers/bars are anything — the AR-invariant compare passes on *declared* values that need not match the *ink*. Tier B would catch the box-pixel mismatch, but **Tier B excludes nothing here that helps**: the attacker simply also paints the boxes correctly (they're cheap to copy from the target raster) — what they cannot cheaply produce is a *real input* with those fingerprint bits, but the SVG engine never demands a real input, it reads the declared attributes. The result: an SVG whose declared render-model equals the target's, assembled by hand with no preimage, passes value-level compare.

**Document grounding.** Brief decision B (auto-detect medium), C-SVG (value-level, AR-invariant fingerprint-derived fields, reconstruct-from-cell-texts); spec *SVG profile* (recover surround bits/markers from attributes, not geometry — "without measuring pixel geometry"); UX/auto-detection pool item (polyglots, "an SVG that rasterizes differently than its render model"); threat-model secondary win "renders differently under encoding form" generalized to *medium* form.

**Recommended fix.** (1) Never auto-route on ambiguous magic: if a file satisfies more than one medium sniff, refuse and ask the user, or force the most conservative (raster, non-EQUAL) engine. (2) The SVG value-engine must **not** trust `data-*` fingerprint attributes from an untrusted reference — it must *recompute* the fingerprint-derived fields from the recovered cell text (which requires a complete ≤512 core, i.e. fall back to UNKNOWN for >512-bit references it cannot recompute) and verify the declared attributes against ink via a non-text-excluded raster pass. A reference whose declared model can't be independently recomputed from its own claimed input must not reach IDENTICAL/NO-DIFFERENCE.

### F7 — MEDIUM: False-DIFFERENT as a denial / social-engineering lever (decisions C-raster, C-SVG, F).

**Attack scenario (T2/T4).** The design treats DIFFERENT as "definitive; any single mismatch proves inequality with certainty" (decision A). But the *machine's* notion of a mismatch can be manufactured by rendering, not by a real value difference. The cross-encoding non-invariance the design inherits (spec *Why the fingerprint hashes text*; threat-model accepted risk) means the *same value* in two encodings yields DIFFERENT. An attacker who controls the reference can deliberately present the legitimate value in a non-matching encoding (hex vs base64 of the same digest; dashed-vs-undashed handled, but base32-vs-base64 not) to make the tool shout **DIFFERENT** on two genuinely-equal values — a denial/erosion-of-trust outcome, and worse, a *training* outcome: a user habituated to "the tool says DIFFERENT but they're actually the same, ignore it" is being conditioned to dismiss the one verdict the design calls definitive. Likewise, F4's near-lossless-probe variant forces UNKNOWN at will, denying the user a verdict and pushing them to a manual walk where F3/F8 attacks live.

**Document grounding.** Decision A (DIFFERENT definitive); spec *Why the fingerprint hashes text* + threat-model *Accepted risks* (cross-encoding non-invariance → same value renders different); brief raster-engine pool item "a *false DIFFERENT* becomes a denial/social-engineering lever."

**Recommended fix.** When the text engine *can* recover both cores, detect cross-encoding equivalence at the *decoded-value* level as an advisory ("these encode the same bytes but entviz compares canonical text; verify the encoding is the one you expect") rather than a bare DIFFERENT — so a user is never trained that DIFFERENT means "ignore me." Surface encoding/medium mismatch as its own state, distinct from value-DIFFERENT.

### F8 — MEDIUM: The focus-ring overlay (decision D, G) can be made to misalign so the human checks the wrong cell, and AR re-render (C-SVG) can make a mismatch look aligned.

**Attack scenario (T2).** Decision D draws the focus ring "around (never over)" a feature as "an ephemeral overlay *off* the entviz SVG (closed profile preserved)." Decision G insists the displayed entviz is the unmodified conformant artifact. The overlay is positioned in the *host page's* coordinate space relative to the entviz's `viewBox`/`width`/`height`. But a conformant entviz is **responsively scaled** (spec *Responsive embedding*: `width="100%"`, `viewBox`-driven), and a malicious embedding (T2 controls surrounding CSS) can apply a `transform: scale()`, a non-uniform `transform: matrix()`, a CSS zoom, or a sub-pixel translate to the entviz element *that the overlay layer does not inherit* — so the ring drawn "around cell 7" lands around cell 8, or between cells, at certain DPIs/zooms. The human dutifully reports [Matches] on the cell the ring *appears* to indicate, which is the attacker's matching decoy cell, not the differing one. Because the overlay is deliberately *off* the SVG (to preserve closed profile), it cannot be pixel-locked to the SVG's internal cell geometry across an attacker-controlled transform — the closed-profile virtue (G) is exactly what prevents the ring from being authoritatively anchored.

For **AR re-render** (C-SVG "re-render ours to the reference grid only for visual alignment"): divergent rasterization (sub-pixel AA, fractional cell widths after AR scaling) can make a true *mismatch* in a cell's surround/nucleus visually align (or a true match look off), biasing the [Matches]/[Differs] call. A reference with an awkward AR (e.g. `97:100`) forces fractional `cell_width` and AA blur that smears a 1-bit surround difference below visibility.

**Document grounding.** Brief decision D (focus ring around/never over, ephemeral off-SVG overlay), G (closed profile, unmodified artifact), C-SVG (AR-independent verdict, re-render for visual alignment); spec *Responsive embedding* (viewBox scaling), *bounding rect* (`crispEdges` borders — but cells are not crisp-rendered, so AA applies); threat-model T2 (surrounding CSS, scale). Lens emphasis bullets on overlay alignment and AR re-render.

**Recommended fix.** Render the comparison entviz in a tool-controlled iframe/shadow-DOM at a *fixed, integer* scale the host cannot transform, and anchor the focus ring in the *same* coordinate system as the entviz (a sibling inside the controlled container, not the host page), re-measuring cell geometry from the live laid-out SVG rather than from `viewBox` math. For AR re-render, never let the *verdict* depend on the visually-aligned raster — keep the visual alignment strictly cosmetic and label it "visual aid, not the basis of the verdict," and render both sides at a scale where one surround box is ≥ a few CSS px to keep 1-bit differences above AA blur.

### F9 — LOW/MEDIUM: Quartile-mark and color-bar-letter glyph forensics — small discrete marks that the raster engine samples are AA-fragile.

**Attack scenario (T2/T4).** The discrete landmarks the habituated user (T6) and the CVD user (§5.4) rely on — color-bar *letters* (`w/g/r/b/k`), quartile *triangle orientation*, blank-map *plus vs dot* shape — are tiny glyphs/triangles. A color-bar letter on a short band MAY have its top bleed above the band (spec *Color-bar letters*, "the top of the glyph MAY bleed above the band"); the blank-map markers are `marker_radius ≈ 3.5 px` and "MAY overflow its sub-cell" on dense grids (spec *Map rendering*). At screen-share / thumbnail scale, AA can render a quartile triangle's orientation ambiguous (top-left vs top-right at 8 px) and a `plus` indistinguishable from a `dot`. A T4/T2 attacker who forces a small viewport / downscale (threat-model T4: "force a small viewport") collapses exactly the color-independent discrete channels the paper §5.4 leans on for the CVD user — so a CVD user can be walked to false confidence on landmark checks whose shapes have blurred together. The viewer-relative bit accounting (decision D) *credits* these checks at full discrete value while the rendering has degraded them below discriminability.

**Document grounding.** Spec *Color-bar letters* (bleed), *Map rendering* (3.5 px markers, overflow, plus/dot shape distinction), *quartile mark* (orientation-only); paper §5.4 (discrete color-independent checks for CVD), §4.3.7; brief a11y==security pool item ("can a CVD or low-vision user be walked to a false high confidence? Does viewer-relative bit accounting over-credit?"); threat-model T4 (small viewport).

**Recommended fix.** Viewer-relative bit crediting must be gated on *measured rendered size*: do not credit a quartile-orientation or plus/dot-shape check unless the mark subtends enough pixels at the actual display scale to be discriminable (compute from the live SVG's rendered geometry, not nominal). When below threshold, schedule a text check instead (the §5.4 "longer path") rather than crediting a blurred shape.

## CONTRADICTIONS (design vs. spec / paper / threat-model)

1. **"Unmatchable text" (decision D/F) vs. spec's explicit homoglyph admission.** The design anchors soundness in text because cell text is unmatchable. The spec's *Font-family fallback chain and homoglyph risk* section states the opposite at the visual layer: glyphs are platform-dependent and `0`/`O`, `1`/`I`/`l`, `5`/`S` are "visually-confusable in some monospace fonts," and Tier B *excludes* glyph pixels for this reason. The soundness claim is true over characters and contradicted over glyphs — the layer every consumer of the claim actually uses. (F1, F3.)

2. **Decision A reserves IDENTICAL for "the *machine* compared both sides in full (pasted SVG)"** vs. spec's Tier-A/Tier-B split, under which a pasted SVG's *text* is verified only as character content (Tier A) and *never* as pixels (Tier B text-exclusion). "Compared in full" is not what the SVG engine does — it compares declared values, not displayed glyphs. (F1, F6.)

3. **Decision C-SVG "compare at the value level… reconstruct normalized core from cell texts"** vs. spec *SVG profile* requiring checkers to recover surround/marker fields "from these attributes, **not** by measuring box geometry." For a *conformance* checker against a *trusted* golden, attribute-trust is fine; for an *adversarial reference*, the design imports a checker designed for trusted input and uses it on untrusted input — a category error the spec's conformance section never licenses. (F6.)

4. **Decision G "closed profile preserved; unmodified conformant artifact"** vs. decision D's "ephemeral overlay off the entviz." The closed-profile virtue (overlay is *not* in the SVG) is precisely what prevents the focus ring from being authoritatively pixel-locked to internal cell geometry under an attacker-controlled host transform. The two decisions are in tension: you cannot have both a provably-anchored ring and an off-SVG overlay in a host-controlled coordinate space. (F8.)

5. **Decision C raster "fidelity self-probe → trust nucleus/colorbar/ellipse sampling"** vs. spec *Guarantees* (nucleus is a hint, sub-JND gradations vanish) and threat-model *Accepted risk* (nucleus unreliable under reduced color depth). The probe trusts a channel the spec says is untrustworthy as a discriminator the moment the pipeline is lossy. (F4, F5.)

6. **Decision E's `~14-bit, commitment-dropped` seed** vs. paper §5.2's *explicit* statement that without commitment the last mover can steer the seed, and §5.2's "the commitment is what removes the last mover's advantage." The brief claims text-anchoring replaces commitment; the paper never makes that substitution — it presents commitment as load-bearing for the seeded-walk soundness. This is outside my lens but flagged: combined with F3 (text not actually unmatchable at the glyph layer), the substitution is unsafe. (Cross-domain; defer to the protocol-lens reviewer, but my F3 is the input their analysis needs.)

## NOVEL ATTACKS (not in the shared pool)

- **N1 — Invisible-text overpaint (F1 concrete).** A reference SVG cell containing two `<text>` runs: the target's characters painted `fill="<nucleus-bg-hex>"` (zero contrast, invisible) and the attacker's characters painted in the real foreground color. Value-engine reads the *first* text node (target → match); human sees the *second*. Closed-profile checker may allow two `<text>`/`<tspan>` if it only counts element *types*, not per-cell instance limits — and the spec's closed profile says "no extra children inside a cell," which a lax comparison-feature reimplementation may not enforce. **Build it:** copy a golden target SVG, for each cell append `<text fill="#<bg>" ...>TARGETTOK</text>` and recolor the existing text to the attacker token.

- **N2 — `data-*` / ink divergence in the AR-invariant path (F6 concrete).** Hand-author a >512-bit reference SVG: set every `data-surround-bits`, `data-blank-map-min/max`, `data-color-bar-rank/band` to the target's values (copied from a published target render), but paint nothing or paint garbage. The AR-invariant value compare reads attributes → "match"; no preimage required. **Build it:** take the target's golden render model JSON, emit a minimal SVG carrying those exact `data-*` attributes.

- **N3 — Late-binding font swap.** Reference SVG with `<defs><style>@font-face{font-family:"JetBrains Mono";src:url(data:font/woff2;base64,<attacker-font>)}</style></defs>` where the embedded font maps the codepoints of the *attacker's* token to glyphs identical to the *target's* token. Now the `<text>` content is the attacker's value (so if the engine recomputes the fingerprint it sees the attacker value and might flag DIFFERENT — good), BUT the human sees the target's glyphs, and in the *guided walk* the human's [Matches] report overrides nothing. This weaponizes the spec's own `font-family` chain: the attacker just *prepends* their malicious `@font-face` to it. **Build it:** subset a font so `U+0061..` (the attacker token's chars) carry the target token's outlines; embed as base64 woff2 in a `<style>` the closed-profile check must strip.

- **N4 — Probe-region steganography for forced-UNKNOWN denial.** Encode "bail" by making *one* probed border pixel `#7f7f7f` instead of `#808080` (1 LSB off, invisible to humans), deterministically forcing the fidelity probe's "degraded → bail to human" branch on demand. Lets an attacker *choose* when the machine refuses, herding users into the manual-walk attack surface (F3/F8) at will. **Build it:** flip one byte in the border row of an otherwise-perfect target raster.

## LENS VERDICT

**Sound within my lens:**
- The **raster engine's hard ceiling** ("DIFFERENT or UNKNOWN, never EQUAL," decision C) is the right call and resists my "force EQUAL from a raster" attacks — *provided* UNKNOWN is made to carry **zero** forward trust (it currently does not; F4). The fail-safe asymmetry (raster can't bless) is the strongest part of the design.
- Drawing the focus ring **around, not over** the feature (D) is correct in intent — it just isn't *anchorable* as specified (F8).

**Broken / unsound within my lens (need rework before this feature ships):**
- **The SVG value-engine trusting `<text>`/`data-*` from an untrusted reference (F1, F6, N1, N2).** This is the critical break: it lets a hand-authored SVG with **no preimage** reach NO-DIFFERENCE/IDENTICAL. The conformance checker was built for *trusted golden* input; reusing it on *adversarial* input is a category error. Must rework to recompute fingerprint-derived fields from the claimed core and verify declared values against non-text-excluded ink.
- **The fidelity self-probe's whole-image trust inference (F4, F5, N4).** Disjoint exact probe regions do not certify the un-probed, un-OCR'd text region; lossy pipelines erase the disproof channel. Must rework so a passed probe credits **no** bits and UNKNOWN resets the walk.
- **Auto-detect routing (F2, F6).** Must gate strict closed-profile + self-containment validation *before* value extraction and refuse ambiguous polyglots.

**My ruling on the two load-bearing questions:**

- **Does anchoring soundness in the text channel survive attack? NO — not as specified.** The claim "text is unmatchable" is sound over the *character alphabet* but the comparison feature consumes it over *glyphs and over attacker-supplied `<text>` bytes*, and the spec itself supplies the gap (platform fonts, Tier-B text exclusion, homoglyph admission). At the **machine** layer the text anchor is defeated by `<text>`/glyph divergence (F1, N1, N3); at the **human** layer it is defeated by homoglyph confusables across the fallback chain (F3). The anchor holds only if the verification tool (a) renders security-critical text in an *embedded pinned* high-disambiguation font, (b) disambiguates homoglyph characters in the spoken/visual protocol, and (c) verifies displayed glyphs against `<text>` content for SVG references rather than trusting the character stream. With those three additions it can be made to survive; without them, decisions C-SVG, D, and F do not.

- **Does dropping cryptographic commitment survive? Not on my lens's evidence.** The brief justifies dropping commitment by leaning entirely on "text is unmatchable." My F3 shows that premise is false at the glyph layer, and the paper §5.2 independently insists commitment is what removes the last-mover steering advantage. Since the substitution's sole support (unmatchable text) is the thing I broke, the commitment drop inherits the break. I defer the protocol mechanics to the seed-mechanics reviewer, but flag that **their analysis must not assume text is unmatchable** — feed them F3.

**Net:** The feature's fail-safe instincts (raster can't bless; ring around-not-over; DIFFERENT is definitive) are good, but its two positive-verdict machine paths (SVG-IDENTICAL via trusted `<text>`/`data-*`; raster-UNKNOWN that forwards trust) and its human-text anchor are all breakable by a rendering/forensics attacker with no preimage. **Rework required before any path is allowed to reach IDENTICAL or to credit text bits.**
