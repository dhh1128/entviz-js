## Executive Summary

The three-component split is **mostly artificial as a lifecycle model but load-bearing at exactly one seam**. Pill, static, and compare are already *one object revealing itself in stages* in the code: `<EntvizPill>` already composes a full `<Entviz controls>` inside its expanded popover (`EntvizPill.ts:441`), and `<EntvizCompare>` already renders pristine `<Entviz>` panels driven by the *same* deterministic render inputs (`value/targetAr/fontSizePt/note`). The pill→static seam is therefore already unified — the only defect there is that expand *enlarges without teaching*, and the playground presents the three as sibling demo boxes (`App.tsx:154–172`) rather than as one arc, which is what plants the "artificial distinction" impression in the maintainer's mind. **The single biggest smoothness win** is to make *acquiring a reference the sole doorway into compare* — turn today's separately-wired `<EntvizCompare>` into a discoverable next step from the expanded view, gated by the natural friction that a comparison *requires the user to go get a second thing*. **The one seam that most needs a deliberate gate is static→compare**: it is the recognition→verification boundary, and it must never be crossable by a slide or a hover — the user must *choose to verify* by *supplying a reference*, which is itself the gate (comparison-design.md §5; F-8). The pill boundary and the closed-profile boundary stay hard; the demo-box boundary dissolves.

## Recommended state model

**One object, four disclosure states, driven by one deterministic render-input tuple** `{value, targetAr, fontSizePt, note}` that is *invariant across the whole arc* — that tuple is what tells the user "same object" (§D below). The entviz glyph is the spec-locked anchor and never changes shape across states; only the chrome around it changes to signal "looking" vs. "verifying".

```
                       ┌──────────────────────────────────────────────────────────┐
                       │  same {value, targetAr, fontSizePt, note} throughout        │
                       │  the glyph is the constant anchor; chrome is what changes    │
                       └──────────────────────────────────────────────────────────┘

     click / Enter / Space            "Compare…" (needs a reference)
     ─────────────────────►           ──────────────────────────────►
 ┌────────┐               ┌────────────┐                        ┌──────────────────────┐
 │  PILL  │               │   STATIC    │                        │       COMPARE         │
 │ badge+ │◄──────────────│  (expanded) │◄───────────────────────│  Yours │ Reference     │
 │ type   │   Esc / out   │ <Entviz     │   Esc / "done"/close    │  (two <Entviz>)       │
 └────────┘               │  controls>  │                        │                       │
   recognition            │  + copy     │   ── the GATE lives ──► │  ┌─ verdict machine ─┐ │
   only                   └────────────┘      here, and IS         │  │ PENDING           │ │
   (no equality)                │              "acquire a ref"      │  │  → NO-DIFFERENCE  │ │
                                 │                                  │  │  → DIFFERENT (⊥)  │ │
                            recognition                             │  │  → IDENTICAL      │ │
                            (locate/look/copy)                      │  │    (machine only) │ │
                                                                    │  └───────────────────┘ │
                                                                    │  tab: [Compare] [Voice] │
                                                                    └──────────────────────┘
                                                                          verification

 Legal transitions:
   PILL → STATIC        : click/Enter/Space (existing onClick → openExpand, EntvizPill.ts:350)
   STATIC → PILL        : Esc / click-outside (existing, EntvizPill.ts:221)
   STATIC → COMPARE     : explicit "Compare…" affordance THAT REQUIRES A REFERENCE (new; the gate)
   COMPARE → STATIC     : close/back (returns to the single-figure look)
   PILL  → COMPARE      : *disallowed as a direct edge* (must pass through STATIC) — see boundary table
   COMPARE(verdict) machine: PENDING → {NO-DIFFERENCE | DIFFERENT | IDENTICAL}, per §3/§14.6
       — NO-DIFFERENCE is affirmative-but-not-terminal; only DIFFERENT and IDENTICAL are terminal.
```

### Boundary-by-boundary keep/dissolve

| Boundary | Verdict | Justification |
|---|---|---|
| PILL vs STATIC | **Dissolve (already dissolved in code)** | The pill *is* the collapsed form of the same figure; it already expands to a real `<Entviz controls>` (EntvizPill.ts:441). Both afford only *recognition* (locate/look/copy) — no equality decision crosses this line, so smoothing it costs no security. It is one object collapsed vs. open. |
| STATIC vs COMPARE | **Keep — this is the only load-bearing seam** | This is the recognition→verification line (pill-design.md §2; comparison-design.md §2.1). Looking at one figure and *checking two figures against each other* are categorically different acts. A frictionless slide from one to the other trains glance-equivalence (the exact heuristic §3.3 forbids). The gate is *supplying a reference* — a real acquisition act, not a click. |
| Direct PILL → COMPARE edge | **Keep it disallowed** | A pill cannot become a verdict surface without first becoming a *readable* figure. Allowing pill→compare directly would invite "compare two pills," which have zero identity bits by design (pill-design.md §3.1) — a verdict from that would be meaningless-yet-reassuring. Route compare through STATIC always. |
| The closed-profile boundary (chrome-outside-the-SVG) | **Keep — inviolable** | Every state must contain an *unmodified* entviz; all badges, toolbars, tabs, verdict chips, focus rings live outside/below the SVG (pill-design.md §2, §7; comparison-design.md §7.1). This is orthogonal to the lifecycle and never bends for smoothness. |
| The three demo boxes in the playground | **Dissolve** | `App.tsx:154–172` renders pill, static, and compare as three unrelated sibling `<div>`s. That *presentation* is what makes the split look artificial. Re-presenting them as one flow (§Phased build, slice 0) is pure UX with zero security cost. |

Where `<EntvizCompare>`'s four-state machine nests: it lives **entirely inside the COMPARE state** and is *unreachable* from PILL or STATIC. That containment is the security property — the verdict machine (and its `=`/green) can only be entered by the deliberate reference-acquisition act.

## Seam 1 — pill → static

- **Affordance / motion / copy.** *Keep the existing model, add teaching.* The pill stays a `<button>` whose click/Enter/Space opens the non-modal, viewport-aware popover anchored to it (EntvizPill.ts:345–366, 425–447); hover/focus shows only cursor + tooltip *"View visualization"* and never auto-pops (§3.5) — that restraint is correct and stays. Two concrete deltas, both chrome-only:
  1. **Make expand teach, not just enlarge.** Today the popover is a bare `<Entviz controls>` plus an SR-only channel description (EntvizPill.ts:441–444). Add a one-line, localized *header* strip *above* the figure in the popover chrome (never on the SVG): *"This is the full visualization. Read the cells to check a value — a glance can't."* This converts the enlarge into a mental-model moment (comparison-design.md §14.8, idea 1) at the exact instant the user first sees the discrete channels. It is the cheapest place to plant "recognition ≠ verification" before the user ever reaches compare.
  2. **Motion = "grow from the pill," not "appear elsewhere."** The popover already anchors to the pill via `placeFloater` (EntvizPill.ts:86–98). Add a short (~120ms) scale/opacity transition originating at the pill's box so the figure visibly *is the pill, opened* — reinforcing "same object." Respect `prefers-reduced-motion`. No layout reflow (the popover is `position:fixed`, EntvizPill.ts:126), so prose/table/code-line contexts are unaffected.
- **Robustness (asked in §B).** `position:fixed` + `placeFloater` already handle flip-above-when-no-room and horizontal viewport clamping (EntvizPill.ts:92–97), and RTL right-aligns the floater (`rtl` branch, :95). Inside `overflow:hidden` containers the fixed popover escapes the clip correctly. Small-screen: the popover can exceed a narrow viewport; recommend a `maxWidth: min(92vw, …)` clamp on the popover chrome (CSS var, no API change). At RTL, chrome mirrors, the glyph never does (pill-design.md §8, non-negotiable 2) — already honored.
- **Security contract honored by:** *closed-profile boundary* — the teaching header and the transition are chrome outside the SVG; the figure inside the popover remains an unmodified `<Entviz>` (EntvizPill.ts:441). *Recognition ≠ verification* — expand still affords only look/copy; the header actively *inoculates* against treating the look as a check. No equality decision is introduced at this seam.
- **Anchors:** EntvizPill.ts:345–366 (pill button), :425–447 (popover = `<Entviz controls>` + SR desc), :86–98/:126–155 (`placeFloater`/`useFloating`), :350 (click→openExpand), §3.5, §4, §14.8.

## Seam 2 — static → compare (the hard one)

- **Affordance / motion / copy.** Add **one** affordance to the expanded (STATIC) popover chrome: a low-emphasis text button beneath the copy actions, labeled **"Compare against a reference…"** (the trailing ellipsis signals "this opens a task, and needs input from you" — it is not a verdict-in-one-click). Clicking it does **not** show a verdict and does **not** auto-place a second figure. It transitions the popover into the COMPARE state where **"Yours" is the figure you were already looking at**, and the **Reference panel is an empty, figure-sized drop/paste slot** — exactly today's `placeholderBox` (`EntvizCompare.ts:436`, "drop hint"). The verdict chip stays *absent while pending* (already the behavior: the chip is omitted for `pending`, EntvizCompare.ts:455). **The gate is the empty slot itself**: nothing affirmative can happen until the user *goes and acquires a reference* (paste / drop / file / URL-fetch, §5). Acquiring a second, independent artifact is a real-world act with real cost — that is the load-bearing friction, and we lean into it rather than sanding it off (persona §C: "it requires acquiring a reference, which is itself the natural gate").
  - **Continuity of motion:** "Yours" should *slide from center to the left panel* as the Reference slot opens on the right, so the user sees their one figure *make room for a second* — not a fresh screen. This is why the pill's popover is deliberately non-modal (pill-design.md §4: "the deferred comparison work wants two open side-by-side, which a modal forbids"). The design pays off that decision.
  - **Copy that sets the verdict mental model** (comparison-design.md §14.8, §15.10): when the COMPARE state opens with no reference yet, the chrome reads *"Comparing checks whether your value equals a reference you supply. It can't tell you the reference is trustworthy — that's for you to judge."* This front-loads equality-soundness ≠ reference-authenticity (F-8) *before* any green appears.
- **Discoverable but still deliberate (the tension, persona §C).** Discoverability is satisfied — compare is now reachable in-flow instead of requiring the host to wire a separate component. Deliberateness is preserved because the *transition reveals a task, not a result*: the user must still choose a reference, and the affirmative verdict is still gated behind the verdict machine's rules. The smoothness we add is *incidental-friction removal* (no more hand-wiring, no dead end at the popover); the friction we keep is *load-bearing* (you must acquire a reference; a peek never blesses).
- **How the asymmetry & anti-habituation survive the smoother entry.** The smoother *entry* touches only reference acquisition; it does **not** touch the verdict machine, which already enforces: `=`/green reserved for machine-IDENTICAL (EntvizCompare.ts:172, `chipFor`); "no difference found" copy for the human path, never certainty (§14.6, §14.8); a peek/spot-check stops at PENDING and cannot reach an affirmative (§14.4, F-4); unpredictable check order + probes as anti-habituation (§14.2/§14.7). Because compare is a *distinct state entered by a distinct act*, a user cannot back-slide into "these look the same → same" — they entered a surface whose entire framing is "a visible difference means reject" (§2, §14.8, idea 2).
- **Security contract honored by:** *Recognition ≠ verification* — the transition surfaces a *task requiring a reference*, not a verdict; looking (STATIC) and verifying (COMPARE) stay categorically separate states. *Same/different asymmetry* — untouched: the verdict machine still reserves `=`/green for machine-IDENTICAL and frames the human affirmative as coverage, never certainty (EntvizCompare.ts:172; §14.6). *Anti-habituation* — untouched: unpredictable order + peek-caps-at-PENDING remain in the walk (§14.2/§14.4/§14.7); the entry change adds no new rubber-stampable affirmative. *Closed-profile boundary* — the second figure is another pristine `<Entviz>` panel; the verdict chip, tabs, and focus ring are chrome outside both SVGs (EntvizCompare.ts:401–448, §7.1).
- **Anchors:** EntvizCompare.ts:436 (empty reference slot = the gate), :455 (chip omitted while pending), :172 (`=`/green only for machine-IDENTICAL), :401–448 (Yours|Reference panels), :467–487 (walk launch: peek vs. Complete), §2, §5, §14.4, §14.6, §14.8, §15.10; F-8, F-3, F-4; adjudication Tier-1 (S6/S16 coverage-not-thermometer), Tier-3 (deny affirmative on a peek).

## Anti-patterns (smooth but unsafe — do NOT)

- **Auto-side-by-side on pill hover / an on-pill "compare" glance.** → violates *recognition ≠ verification* and the *pill's zero-identity-bit* rule (pill-design.md §3.1/§3.3). Two pills carry no identity, and putting them side-by-side trains glance-equivalence. Compare must route through STATIC and require a supplied reference.
- **A direct PILL → COMPARE edge (skip the expanded look).** → violates the same two contracts; a pill can never become a verdict surface without first becoming a *readable* figure.
- **Auto-fetching or auto-detecting a reference "to be helpful" the moment compare opens.** → violates *equality-soundness ≠ reference-authenticity* (F-8) and the fail-closed rule (comparison-design.md §5; S12). The reference must be a *user-chosen* artifact with visible provenance and origin-shown-before-fetch (EntvizCompare.ts:306–315 already shows origin first — keep it).
- **Showing any green/`=`/"identical" wording on a human walk, or a `1-in-2^N` thermometer.** → violates the *same/different asymmetry* and the coverage-not-probability ruling (§3, §14.6; adjudication Tier-1 S6). Reserve `=`/green for machine-IDENTICAL only (EntvizCompare.ts:172); the human affirmative is "no difference found across what you checked."
- **Letting a peek/quick spot-check land on an affirmative verdict "so the flow feels complete."** → violates *anti-habituation* (F-4, N-1; §14.4): a peek must cap at PENDING with "a sanity look, not a verification," or it trains the click-through reflex that transfers to Strong/Complete sessions.
- **A satisfying "verified!" success animation on NO-DIFFERENCE.** → violates the asymmetry (F-3): users learn the *gestalt of success* and stop parsing the qualifier. Keep the affirmative visually sober; save any celebratory treatment (if any) strictly for machine-IDENTICAL.
- **Collapsing the voice ceremony into the machine flow "to reduce tabs."** → violates the ceremony's distinct guarantee framing (§15.8/§15.10). Voice stays a peer *tab* with its own expectation-setter (equality-not-identity), already implemented (EntvizCompare.ts:494–508, 533–538).
- **Drawing the "Compare…" affordance, the teaching header, or the transition *on* the entviz SVG.** → violates the *closed-profile boundary* (pill-design.md §2/§7). All lifecycle chrome lives outside/below the glyph.

## API delta

Composition-first; **no new wrapper component is warranted yet.** The three components already share the deterministic render-input tuple and already nest (`EntvizPill` → `Entviz`; `EntvizCompare` → `Entviz`). The lifecycle is a *state* concern, not a new artifact.

**Reused as-is (no change):**
- `EntvizPill.onExpand` (EntvizPill.ts:60, :246) — already fires on pill→static; the host's existing hook to observe entry into STATIC.
- `EntvizCompare.reference: { kind: "text" | "svg"; data: string }` (EntvizCompare.ts:38) — the host-controlled reference path; unchanged. Also its `layout` (`"side-by-side" | "stacked" | "auto"`, :43), `onVerdict` (:48), and the shared size/shape controls that drive both panels (:219–222).
- The verdict machine, walk launch, and voice tab in `EntvizCompare` — untouched by this proposal.

**New (small, chrome-only):**
1. **`EntvizPill` new optional prop `onCompare?: () => void`** — mirrors `onExpand`. When provided, the expanded popover renders the **"Compare against a reference…"** affordance in its chrome (Seam 2). When omitted, the affordance is absent and behavior is exactly today's (host opt-in preserves the property split — a host that only wants recognition/look never surfaces compare). This is the single load-bearing new API.
2. **`EntvizPill` new optional prop `showCompareAffordance?: boolean` (default: `true` iff `onCompare` given)** — lets a host keep `onCompare` telemetry but suppress the in-popover button if they wire their own entry point.
3. **A new message-catalog key** for the teaching header (Seam 1) and the compare-affordance label + the pre-reference framing copy (Seam 2), added to the existing `Messages`/`CompareMessages` catalogs (pill-messages.ts / compare-messages.ts). Localized like all other chrome (pill-design.md §8). *No value/comparison-text is ever localized* (§8 non-negotiable 1).
4. **CSS custom properties** for the new chrome (`--entviz-pill-teach-*`, transition timing) so hosts restyle without forking (matches the existing `--entviz-pill-*` convention, EntvizPill.ts:185/:289).

**Explicitly NOT added:** no `<EntvizLifecycle>` wrapper, no `mode` prop that flattens the states, no direct `EntvizPill → EntvizCompare` prop bridge. The states stay distinct components composed by the host; the *only* new coupling is the opt-in `onCompare` callback that lets a pill *offer* the transition its expanded child already renders the destination for.

## Phased build handoff (glyph-untouched, TDD + milestone slices)

- **Slice 0 (UX only, zero security surface) — Re-present the playground as one arc.** Replace the three sibling boxes (`App.tsx:154–172`) with a single narrative: a pill in prose → "click to open" → the expanded figure → "have something to check it against?" → compare. This is the demonstration that the split is a *lifecycle*, not three widgets. Tests: none needed beyond render smoke; it is a demo. **Lowest risk.**
- **Slice 1 — Teaching header + grow-from-pill transition (Seam 1).** Add the localized header to the popover chrome and the `prefers-reduced-motion`-aware scale/opacity transition; add the popover `maxWidth` clamp. TDD: unit-test the message resolution and that the SVG is unchanged (closed-profile assertion — assert the popover's `<Entviz>` markup is byte-identical with/without the header). **Low risk** (chrome only).
- **Slice 2 — `onCompare` + in-popover "Compare against a reference…" affordance (Seam 2 entry).** Wire the opt-in prop and render the button; on click, transition the popover into the COMPARE state with "Yours" populated and the Reference slot empty. TDD: the affordance is absent without `onCompare`; clicking it produces the pending compare surface with **no verdict** and an empty reference slot (assert chip is omitted while pending, mirroring EntvizCompare.ts:455). **Medium risk** — this is the state boundary; the test must prove *no affirmative verdict is reachable at open*.
- **Slice 3 — Continuity motion (Yours slides center→left as Reference opens) + pre-reference framing copy.** Purely presentational + one message key. TDD: framing copy present before any reference is supplied; asserts equality-not-identity wording (§15.10) renders. **Low risk.**
- **Slice 4 (RISKIEST TO THE SECURITY POSTURE) — the anti-pattern regression guardrails.** Add explicit tests that codify the "do NOT" list so a later refactor can't quietly cross the gate: (a) no direct pill→compare path exists (there is no code edge from the pill to a verdict without a supplied reference); (b) a peek/spot-check cannot reach an affirmative verdict (already true — §14.4 — but assert it *from the lifecycle entry* so the smoother entry can't regress it); (c) `=`/green appears only for machine-IDENTICAL (assert `chipFor` behavior from the new entry, EntvizCompare.ts:172); (d) the reference is never auto-acquired on compare-open (no fetch fires without user action). **This slice is where a well-meaning "make it smoother" change would erode the contracts, so its tests are the load-bearing safety net — build it before shipping Slice 2/3 to users.**

Ordering note: Slices 0–1 can ship independently for immediate value. Slice 4's guard tests should land *with or before* Slice 2 so the new state boundary is born fenced.

## Open questions (maintainer's call)

1. **Should the "Compare against a reference…" affordance be on by default when `onCompare` is provided, or require the host to also set `showCompareAffordance`?**
   *Recommended default:* on when `onCompare` is given (opt-in is the callback itself). Rationale: a host that wants recognition-only simply never passes `onCompare`, preserving the property split; a host that wants compare gets discoverability for free. Reconsider only if telemetry-without-UI turns out to be a common host need.
2. **Exact wording of the pill→static teaching header and the pre-reference compare framing (§14.8 territory — deliberately the maintainer's call).**
   *Recommended defaults* (localizable): header — *"This is the full visualization. Read the cells to check a value — a glance can't."*; pre-reference — *"Comparing checks whether your value equals a reference you supply. It can't tell you the reference is trustworthy."* These instill idea 1 (a glance can miss) and equality-soundness ≠ reference-authenticity (F-8, §15.10). Wording, not mechanism, is what's open.
3. **Is compare ever legitimately *pill-initiated* for a first-party host that renders its own trusted label beside the pill (pill-design.md §3.2)?**
   *Recommended default:* No — always route through STATIC (the disallowed-direct-edge boundary). A first-party host that wants a one-click "verify this" can call `onExpand`→auto-`onCompare` in *their* code, but the component never collapses the look-then-verify sequence itself. Keep the two acts distinct in the primitive; let hosts compose shortcuts at their own trust context.
4. **Does the "grow-from-pill" motion risk implying the popover *is* verification-grade at small sizes?**
   *Recommended default:* No motion change to meaning; but gate the compare affordance's visibility on the expanded figure being at/above a legible size (reuse the walk's discriminability-size logic, comparison-design.md §7.3) so a user can't jump to compare from a sub-legible figure. Flag for the build pass to confirm against the calibration thresholds.
