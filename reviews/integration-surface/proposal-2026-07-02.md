## 1. Executive summary

**Recommended shape: keep the ergonomic per-component callbacks AND add one typed firehose `onEvent(e: EntvizEvent)` to every component, emitting a discriminated union that mirrors the whole disclosure lifecycle (Cite · Visualize · Compare) and the comparison journey (acquisition · fetch · medium · verdict · walk · voice · display).** Today the surface is a scatter of ad-hoc callbacks — `onError`/`onResize`/`onReshape` on `<Entviz>` (`Entviz.ts:25,35,38`), `onExpand`/`onCompare`/`onCopy`/`onError` on `<EntvizPill>` (`EntvizPill.ts:61,66,70,71`), a lone `onVerdict` on `<EntvizCompare>` (`EntvizCompare.ts:49`), and `onStep`/`onComplete` on the two verification components (`EntvizWalk.ts:67,68`; `EntvizVoiceCompare.ts:58,59`) — and, critically, **`<EntvizCompare>` forwards almost none of its own internal life to the host**: reference acquisition/clearing, medium detection, the fetch lifecycle, decode/read failures, the secret-detected warning, tab changes, walk-mode selection, and the nested walk/voice progress all happen inside `EntvizCompare` (`EntvizCompare.ts:207–320, 476–508`) with **no callback out**. A host that drops the comparator into a real app is blind to everything except the final `Verdict`.

**Biggest DevX win:** the firehose lets a host observe, log, and build telemetry/analytics/custom affordances over the *entire* journey — especially the comparison journey, which is currently a black box between "user opened it" and "onVerdict fired" — without forking the component. It also future-proofs: one `onEvent` absorbs new event kinds without new props.

**Sharpest security risk I am introducing:** a firehose is the *ideal* exfiltration channel. entviz treats the value and the reference as potentially **secret** (`comparison-design.md` §5; `looksLikeSecret` at `EntvizCompare.ts:95`), and confidentiality is out of scope by design. If any event payload carries the pasted/dropped/typed/fetched **bytes**, the host's ordinary `onEvent`-to-logger wiring silently captures private keys and seed phrases. **My mitigation is a hard default: every event payload is metadata-only (medium, provenance, origin, byte-length, verdict *state*, timings, coverage bits). Raw content is available only behind one explicit, loudly-named opt-in prop (`unsafeIncludeContent`), off by default, and even then never on the pill.** The second-sharpest risk is that a rich, cancelable event surface could let a host force a verdict or disable a safety; I make the verdict/verification and secret-warning events **notify-only, never cancelable**, and gate acquisition/fetch controls so they can restrict but never *forge* an affirmative (§4, §5).

## 2. Event catalog

### 2.1 The `EntvizEvent` union and the `onEvent` contract

Every component gains an optional `onEvent?: (e: EntvizEvent) => void`. It is a **notify-only firehose**: the return value is ignored, it never blocks rendering, and it fires *in addition to* (not instead of) the existing specific callbacks, which are retained for ergonomics. Each event is an object with a discriminant `type`, a monotonic `seq`, a `ts` (ms epoch), a `source` (which component emitted it), and a `sensitivity` tag echoed into the payload so a host's logger can filter by policy without knowing every `type`. Payloads are **object-shaped** so fields can be added without breaking callers (persona §B future-proofing).

```ts
type EntvizSensitivity = "safe" | "network" | "content";
type EntvizSource = "entviz" | "pill" | "compare" | "walk" | "voice";

interface EntvizEventBase {
  seq: number;                 // monotonic within a component instance
  ts: number;                  // Date.now() at emit
  source: EntvizSource;
  sensitivity: EntvizSensitivity;
}

type EntvizEvent =
  // ---- lifecycle (Entviz / Pill) ----
  | (EntvizEventBase & { type: "render.error"; message: string })                                   // safe
  | (EntvizEventBase & { type: "disclosure.change"; state: "pill" | "visualize" | "compare";
                          prev: "pill" | "visualize" | "compare" })                                  // safe
  | (EntvizEventBase & { type: "copy"; kind: CopyKind; ok: boolean; byteLength?: number })           // safe (metadata only — NOT the copied text)
  | (EntvizEventBase & { type: "display.resize"; fontSizePt: number })                               // safe
  | (EntvizEventBase & { type: "display.reshape"; targetAr: number; cols: number; rows: number })    // safe
  | (EntvizEventBase & { type: "display.tab"; tab: "reference" | "voice" })                          // safe
  // ---- acquisition (Compare) ----
  | (EntvizEventBase & { type: "reference.acquired"; provenance: Provenance;
                          medium: "text" | "svg" | "raster" | "ambiguous" | null;
                          byteLength: number; origin?: string; content?: string })                  // safe by default; content only under opt-in
  | (EntvizEventBase & { type: "reference.cleared" })                                                // safe
  | (EntvizEventBase & { type: "reference.mediumDetected";
                          medium: "text" | "svg" | "raster" | "ambiguous"; isUrl: boolean })         // safe
  | (EntvizEventBase & { type: "reference.readError"; reason: string })                              // safe
  | (EntvizEventBase & { type: "secret.detected"; where: "value" | "reference" | "both" })           // safe (a boolean-ish signal; carries NO bytes)
  // ---- fetch (Compare, network) ----
  | (EntvizEventBase & { type: "fetch.start"; origin: string; url?: string })                        // network
  | (EntvizEventBase & { type: "fetch.success"; origin: string; status: number;
                          byteLength: number; durationMs: number })                                  // network
  | (EntvizEventBase & { type: "fetch.error"; origin: string; message: string })                     // network
  // ---- outcome (Compare) ----
  | (EntvizEventBase & { type: "verdict.change"; verdict: VerdictState;
                          medium: string | null; coverageBits?: number; complete?: boolean })        // safe
  // ---- verification (Walk / Voice, forwarded up through Compare) ----
  | (EntvizEventBase & { type: "walk.start"; mode: "spot-check" | "complete" })                       // safe
  | (EntvizEventBase & { type: "walk.step"; feature: string | null; index: number })                 // safe (feature *kind*, e.g. "text"/"ellipse" — never cell bytes)
  | (EntvizEventBase & { type: "walk.complete"; status: WalkStatus })                                 // safe
  | (EntvizEventBase & { type: "voice.start"; mode: "voice-only" | "paste-bind" })                    // safe
  | (EntvizEventBase & { type: "voice.step"; cellAddress: string | null; index: number })             // safe (grid address "row 1, col 2" — a coordinate, not the glyph)
  | (EntvizEventBase & { type: "voice.complete"; status: CeremonyStatus });                           // safe

// VerdictState = "pending" | "different" | "no-difference" | "identical" | "unknown"
```

`onEvent` may throw in host code; components MUST wrap the call in try/catch so a host bug can't break the widget or, worse, wedge it in a state that skips a safety.

That is the same information the closed-profile ring already shows on screen.

## 4. Payload & safety policy

**1. Metadata-not-content is the default, everywhere.** Every event payload carries only: `medium`, `provenance`, `origin`, `byteLength`, `verdict`/`walk`/`voice` *state*, coverage bits, feature *kind*, grid *address*, timings, and error *messages* (renderer/fetch diagnostics). **No event carries the value, the reference bytes, the pasted/typed text, the comparison text, the fetched body, or a cell's glyphs by default.** `content`-tagged fields (`reference.acquired.content`, `fetch.start.url`) exist in the type but are populated only when `unsafeIncludeContent` is set.

**2. The content opt-in is one prop, loudly named, off by default, pill-forbidden.** `unsafeIncludeContent` (§3). Rationale: the value/reference may be a private key or seed phrase (`comparison-design.md` §5; `looksLikeSecret`, `EntvizCompare.ts:95–102`); the firehose is exactly where a host's request logger would capture it. Making content opt-in with a name containing `unsafe` forces the host to consciously accept exfiltration risk, and forbidding it on the pill keeps the recognition-only surface incapable of leaking value bytes even by misconfiguration.

**3. `content`-tagged events, explicitly:** only `reference.acquired.content` and `fetch.start.url`. Everything else is `safe` or `network`. `network`-tagged events (`fetch.*`) carry the **origin** (already shown pre-fetch, `EntvizCompare.ts:375`) and metadata, but **not** the response body; a host that needs the body opts into content and gets it on `reference.acquired` after routing, not raw off the wire.

**4. Fetch guard rails (host `fetchReference` + `allow.url`).** The injected fetcher (a) runs only *after* the origin is displayed to the user (`fetchHint`, `EntvizCompare.ts:375`), preserving origin-before-fetch (`comparison-design.md` §5); (b) receives an `AbortSignal` so the component can cancel; (c) its result is fed through the same `detectMedium` + fail-closed classification (`comparison-design.md` §5) — it cannot hand back a pre-classified "this is IDENTICAL"; (d) `allow.url:false` removes the fetch path entirely for DLP/air-gapped hosts. `fetch.start` is the **one advisory-cancelable event**: a host may call `e.preventDefault?.()` to *block* a fetch (fail-closed, safe direction — blocking can only *deny* network egress, never force a pass). Blocking is honored; there is no event that can *cause* or *upgrade* an affirmative verdict.

**5. Non-cancelable invariants (the security spine).**
- `verdict.change` is **notify-only**. No host code can veto, delay, upgrade, or synthesize a verdict; the verdict machine stays authoritative (`comparison-design.md` §2/§3; `=`/green reserved for machine-IDENTICAL at `EntvizCompare.ts:172–173`).
- `walk.complete`/`voice.complete` are **notify-only**. A host cannot turn a `different`/`inconclusive`/`pending` into `no-difference`, nor a `no-difference` into `identical` (the walk/ceremony cap at NO-DIFFERENCE by construction — `comparison-design.md` §14.6, §15.7).
- `secret.detected` is **notify-only** and fires *regardless* of `secretWarning:"off"`. Suppressing the banner is a host-owned display choice (gated behind an explicit acknowledgment flag); it never suppresses the *signal*.
- `walk.step`/`voice.step` cannot be redirected — a host can't steer which feature/cell is checked via an event return; selection stays with the CSPRNG/authenticator (`comparison-design.md` §14.2/§15.3).

**6. Ordering & idempotency.** Events carry a monotonic `seq`; hosts must tolerate duplicate `verdict.change` with the same state (React may re-emit) and out-of-order delivery across component boundaries (a nested `walk.step` and the parent `verdict.change` are on the same tick). The component de-dupes its own emissions per `(type,payload)` where cheap, but the host must be defensive.

## ⚑ 5. Security-sensitive decisions (handoff to the adversary)

*Each item is phrased as an attackable claim: I propose X; I believe it's safe because Y; the risk if I'm wrong is Z. The adversary attacks these next — I have hidden nothing.*

1. **I propose a firehose `onEvent` that emits across the whole journey.** I believe it's safe because every default payload is metadata-only (§4.1) and the components wrap the call in try/catch. **Risk if I'm wrong:** the firehose is the single richest exfiltration and correlation surface in the library; if *any* payload field I've tagged `safe` is actually value-revealing (see items 2–4), one line of host logging leaks secrets. The whole proposal's safety rests on the metadata/content line being drawn correctly.

2. **I propose `reference.acquired` carries `byteLength` (not content) by default.** I believe it's safe because a length is not the value. **Risk if I'm wrong:** byte-length is an oracle. For a *constrained-format* value (a specific key type, a UUID, a fixed-width hash) the length narrows the type and, combined with `medium` + `secret.detected` + `provenance`, may fingerprint *which* secret was pasted even without its bytes. An adversary should decide whether `byteLength` should be bucketed/omitted for secret-shaped inputs.

3. **I propose `secret.detected` emits the *fact* (and `where`) but no bytes, and is non-cancelable.** I believe it's safe because it lets hosts do DLP/telemetry without the payload being the leak, and non-cancelability keeps the banner from being silenced via a veto. **Risk if I'm wrong:** (a) the event *itself* becomes a "this user just pasted a private key" beacon into whatever the host wires `onEvent` to — a privacy signal that didn't exist before; (b) `looksLikeSecret` (`EntvizCompare.ts:95–102`) is a *heuristic* — false negatives mean the event (and banner) don't fire on a real secret, and a host that trusts `secret.detected` for DLP gets a false sense of coverage. The signal must be documented as best-effort, never a guarantee.

4. **I propose `walk.step`/`voice.step` are `safe` because they carry feature-kind / grid-address, not glyphs.** I believe it's safe because an address is a coordinate, not content, and matches what the on-screen ring already reveals. **Risk if I'm wrong:** the *sequence* of addresses over a full walk is the CSPRNG-selected check order. If that order is observable to a party who can also influence timing, it could weaken the unpredictable-coverage defense (`comparison-design.md` §14.2; F-4/F-5 in `findings-02`). In the single-user case the order is local and this is moot; in any future shared/live-instrumented case, streaming the live check order to a host that relays it is a steering aid. Adversary should confirm the walk order must never leave the local endpoint in the live ceremony.

5. **I propose `unsafeIncludeContent` as the one opt-in that puts raw value/reference bytes into events.** I believe it's safe because it's off by default, named to be un-ignorable, and forbidden on the pill. **Risk if I'm wrong:** it exists at all. Any prop that can be flipped can be flipped by a copy-pasted config, a "temporarily debugging" toggle left on, or a host framework that spreads unknown props. Once on, every `reference.acquired` hands the secret to `onEvent`. The naming and pill-exclusion are the only guardrails; an adversary may argue content should never be an event payload at all and should instead require a separate explicit imperative API call per-read.

6. **I propose host-injected `fetchReference(url, {origin, signal})`.** I believe it's safe because origin-before-fetch still holds, the result is re-classified fail-closed, and it supplies bytes not verdicts. **Risk if I'm wrong:** a host fetcher is an SSRF/exfil primitive I'm formally blessing. It can be pointed at internal metadata endpoints, it sees the URL (which may carry secrets in query/fragment), and a malicious/buggy host fetcher can return attacker-chosen bytes that then flow into the SVG engine — the `comparison-design.md` §6.2 "SVG is attacker-authorable" path. The re-classification and self-consistency checks are what stop this becoming a forged IDENTICAL; if those core checks have a gap, `fetchReference` widens its blast radius.

7. **I propose `allow.url` (and `allow.*`) can turn acquisition methods OFF but never ON beyond the built-ins.** I believe it's safe because restriction is monotonic toward less attack surface. **Risk if I'm wrong:** if the implementation ever reads `allow` to *enable* a path that's otherwise gated (e.g. a desktop typed-path reference, `comparison-design.md` §12), the "restrict-only" invariant breaks and `allow` becomes a capability grant. The adversary should verify no code path treats a missing `allow` key as "enable something new."

8. **I propose `secretWarning:"off"` is possible but requires a second explicit `acknowledgeSecretRiskOwned:true` flag.** I believe it's safe because it forces the host to consciously own the risk rather than fat-finger a single suppress-prop. **Risk if I'm wrong:** the banner is a load-bearing usable-security mitigation (F-9; `comparison-design.md` §5). Any suppression path, however gated, means some hosts will ship it off, and users of *those* hosts paste secrets with no warning. The mitigation degrades to "we told the host to be careful." An adversary may argue the banner must be non-suppressible entirely, with only its *styling* themeable.

9. **I propose `fetch.start` be advisory-cancelable (host may block a fetch).** I believe it's safe because cancellation is fail-closed — it can only *deny* egress. **Risk if I'm wrong:** introducing *any* cancelable event trains hosts (and future maintainers) to expect `preventDefault` semantics, and a later careless addition could make a *verdict* or *secret* event cancelable by pattern-matching. The cancelable/notify-only line must be enforced in the type system (only `fetch.start` gets a `preventDefault`), not by convention, or item 5's guardrail erodes.

10. **I propose generalizing `rng` injection to `<EntvizWalk>` and `<EntvizCompare>`.** I believe it's safe because it defaults to the platform CSPRNG (as `<EntvizVoiceCompare>` already does, `EntvizVoiceCompare.ts:110,127`) and is meant for tests/repro-demos. **Risk if I'm wrong:** a predictable/seeded `rng` shipped to production defeats the unpredictable-order anti-habituation (`comparison-design.md` §14.2) and, in the ceremony, the selection-unpredictability that substitutes for the removed seed (§15.3) — an attacker who knows the sequence pre-forges exactly the sampled cells. This is a foot-gun by configuration. The adversary should decide whether `rng` injection must be compiled out of production builds or at least loudly warn at runtime when non-CSPRNG.

11. **I propose `disclosure.change`/`open`/`onOpenChange` for controlled pill disclosure.** I believe it's safe because disclosure is recognition-only and the recognition→verification gate still requires a *supplied reference* (`ux-disclosure-lifecycle/proposal` Seam 2, boundary table). **Risk if I'm wrong:** if a host can drive `open`→compare programmatically *and* also inject a reference programmatically (via the existing `reference` prop, `EntvizCompare.ts:39`), it can assemble "auto-open + auto-reference + read `verdict.change`" — a headless compare with an attacker-chosen reference and no human in the loop (F-8: equality ≠ reference-authenticity). The events don't *create* this (the `reference` prop already allows a provided reference), but they make it *observable and automatable* end-to-end. The adversary should weigh whether making the full journey scriptable materially eases the attacker-chosen-reference social-engineering path.

12. **I propose `data-*`/`aria-*`/`id` passthrough onto the chrome root.** I believe it's safe because it's filtered to exclude `on*`/`href`/`style`-on-SVG and never touches the entviz `<svg>` (closed profile, `pill-design.md` §2; `Entviz.ts:41–51`). **Risk if I'm wrong:** passthrough is a classic injection vector. If the filter is incomplete, a host (or a prop-spreading framework passing through untrusted attributes) could land an event handler or a styling hook on or near the closed artifact, reintroducing the exact XSS/overlay vector `Entviz.ts:41–51` documents as currently absent. The filter allowlist is security-critical, not cosmetic.

## 6. Open questions (each with a recommended default)

1. **Should content ever be an event payload, or only reachable via an imperative pull API?**
   *Recommended default:* keep `unsafeIncludeContent` (event-push) for DevX parity, but document an imperative `getReference()` ref-handle as the *preferred* path so content retrieval is an explicit act, not an ambient stream. Revisit if the adversary (item 5) rules push-content unacceptable — then drop the event field and keep only the imperative pull.

2. **Should `byteLength` be bucketed or omitted for secret-shaped inputs?**
   *Recommended default:* emit exact `byteLength` for non-secret media; when `secret.detected` is true, coarsen to a bucket (`<32`, `32–64`, `64–128`, `128+`) to blunt the length-oracle of item 2. Cheap, and preserves most telemetry value.

3. **Must a non-CSPRNG `rng` be dev/test-only?**
   *Recommended default:* accept `rng` in all builds but emit a one-time `render.error`-adjacent console warning ("non-default rng — anti-habituation/selection-unpredictability weakened; do not ship") when the injected source isn't the platform CSPRNG. A hard compile-out is safer but breaks Storybook/visual-regression setups that legitimately need determinism.

4. **Firehose only, or keep every legacy scalar callback too?**
   *Recommended default:* keep both indefinitely (source-compat; `onVerdict`, `onCopy`, `onResize`, `onExpand`, `onCompare`, `onStep`, `onComplete` all stay). The firehose is additive. Deprecate nothing yet; re-evaluate at a major version.

5. **Does `disclosure.change` belong on `<Entviz>` (which has no disclosure states) or only on `<EntvizPill>`/`<EntvizCompare>`?**
   *Recommended default:* only where states exist (Pill, Compare). `<Entviz>` emits `display.*`, `copy`, `render.error` but not `disclosure.change`. Keeps the union honest per `source`.

6. **Should `fetch.*`/`reference.*` events fire for a host-*provided* reference (`reference` prop, provenance `"provided"`)?**
   *Recommended default:* fire `reference.acquired` with `provenance:"provided"` (so telemetry is uniform) but **not** `fetch.*` (no network happened). This matches the code path where `provided` short-circuits acquisition (`EntvizCompare.ts:250–251, 325`).

7. **Rail/lifecycle wording aside, should `disclosure.change` use the locked stage names (Cite/Visualize/Compare) or the internal state names (pill/visualize/compare)?**
   *Recommended default:* payload uses stable internal identifiers (`"pill"|"visualize"|"compare"`); the *display* labels (Cite/Reference/Look/Compare — `lock-checklist` §F) stay localizable chrome and never enter the event contract, so a label rename can't break a host's analytics.
