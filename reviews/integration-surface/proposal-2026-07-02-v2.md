## 0. What changed from v1 and why (threat-model correction)

The v1 proposal (`proposal-2026-07-02.md`) and its adversary pass
(`security-adversary-2026-07-02.md`) were **mis-scoped**. They spent almost the
entire caution budget defending **confidentiality of the input** — a
metadata-vs-content firewall, `unsafeIncludeContent`, byte-length oracles,
"secret beacon" telemetry, error-message content channels. Per
`../entviz/docs/threat-model.md` **Out of scope**: *"Confidentiality of the
input. entviz is a comparison aid, not a secrecy primitive."* entviz values are
usually **public** (pubkeys, identifiers, addresses, CIDRs). Keeping value bytes
out of a host logger defends a **non-asset**. That whole apparatus was theater.

The **real** frame (threat-model *Assets*): the ONE primary asset is **the
user's belief that two values are equal (or not)**; the single attacker win is a
**false "same"** (`A ≠ B` concluded same). Secondary asset: **integrity of the
embedded SVG** (no silent alteration, no script/style bleed). **Tier T2** — the
attacker controls the rendering surface (surrounding CSS, fonts, scale) — is
in scope and is exactly where a React component's theming vars, `style`/`className`
passthrough, and `messages` overrides are most exposed.

**Concretely, what this v2 does to v1:**

| v1 apparatus (confidentiality-driven) | v2 disposition |
|---|---|
| `unsafeIncludeContent` push field + imperative-pull debate | **Relaxed** to one light off-by-default `includeContent` flag (Compare only, pill-forbidden). The residual is the corner-case pasted hex/base64 *secret*, adequately covered by the existing secret banner. No `unsafe*` name, no imperative-pull ceremony. |
| `byteLength` bucket-vs-omit for secrets (length oracle) | **Dropped as a concern.** Length is not the asset. Emit it plainly. |
| `secret.detected` "beacon"/DLP framing, non-cancelable | **Kept as a plain event, de-fanged of the beacon/DLP theater.** It's a UX hint, not a confidentiality signal. Non-cancelable stays (a host must not veto the banner). |
| error `message`/`reason` "content channel" REJECT | **Relaxed.** A message echoing the (public) value is not a leak. Keep messages free-form for DevX. |
| `maxSensitivity` in-component cap for exfil | **Dropped.** No exfil asset to cap. |
| Fetch as "SSRF/credential-exfil primitive" (v1's headline) | **Re-scoped.** SSRF/egress is a host-app concern, out of entviz's asset frame. The fetch guard that *matters* is **integrity**: fetched bytes are attacker-authorable and must run the §6.2 gauntlet before they can reach a verdict or the SVG engine — that is the guard, and it already exists in core. |

**What v2 ADDS — the guards that actually matter (all T2 / integrity / false-same):**

1. **Theming/`--entviz-*`/`style`/`className`/`data-*` passthrough that DISTORTS or
   OVERLAYS the glyph, or DESYNCS "yours" vs "reference" size/shape.** This is the
   sharpest real risk and v1 barely touched it. Today an *ancestor* stylesheet
   (T2) can set `--entviz-compare-bad: #1a7f37` and repaint a `≠` verdict in
   verdict-green (`EntvizCompare.ts:193–198`), or set `--entviz-walk-ring: transparent`
   + `--entviz-walk-scrim: transparent` to **erase the walk's attention spotlight**
   (`EntvizWalk.ts:73–74`), or `transform`/`filter`/`mix-blend-mode` one figure to
   desync the pair — no prop needed. v2 hardens the verdict/ring channels against
   this and constrains what passthrough can reach.
2. **`messages` override changing the VERDICT symbol/tone/semantics.** Only
   surrounding prose is localizable; the verdict symbol/tone/scoping-copy is
   authoritative and not override-able.
3. **Controlled disclosure + `reference` + `verdict.change` easing the
   attacker-chosen-reference path.** Provenance + the "equal to THIS reference,
   not trustworthy" scoping copy (§2.4) always render, even under host control.
4. **`rng` injection / leaking the walk/ceremony check ORDER via `*.step`,
   defeating the unpredictable sampling** (§14.2/§15.3/§15.9).
5. **Fetch returning attacker-authorable SVG bytes** — the existing §6.2
   re-classification/self-consistency gauntlet is the guard; a host `fetchReference`
   must not bypass it.

Items retained unchanged from v1 (they were right): the firehose+callbacks shape;
the object-shaped payloads; the notify-only verdict/verification invariants;
`allow` as restrict-only; controlled `open`/`onOpenChange`; `data-*` passthrough
via **allowlist** (the adversary's T-10 fix, which happens to also be the correct
*integrity* fix — a stray `style`/`transform` attribute near the glyph is a
distortion vector, not just an XSS one).

## 1. Executive summary

**Recommended shape (unchanged from v1): keep the ergonomic per-component
callbacks AND add one typed firehose `onEvent(e: EntvizEvent)` to every
component**, emitting a discriminated union mirroring the whole disclosure
lifecycle (Cite · Visualize · Compare) and the comparison journey (acquisition ·
fetch · medium · verdict · walk · voice · display). Today the surface is a
scatter of ad-hoc callbacks — `onError`/`onResize`/`onReshape` on `<Entviz>`
(`Entviz.ts:25,35,38`), `onExpand`/`onCompare`/`onCopy`/`onError` on `<EntvizPill>`
(`EntvizPill.ts:61,66,70,71`), a lone `onVerdict` on `<EntvizCompare>`
(`EntvizCompare.ts:49`), `onStep`/`onComplete` on the two verification components
(`EntvizWalk.ts:67,68`; `EntvizVoiceCompare.ts:58,59`) — and `<EntvizCompare>`
forwards almost none of its own internal life: reference acquisition/clearing
(`EntvizCompare.ts:302–320, 525–535`), medium detection (`:254`), fetch
(`:311–320`), decode/read failures (`:305–308`), tab changes (`:214`), walk-mode
selection (`:210,494`), and nested walk/voice progress. A host is blind to
everything but the final `Verdict`.

**Biggest DevX win:** the firehose makes the *entire* comparison journey —
currently a black box between "opened" and "onVerdict fired" — observable and
loggable without forking, and future-proofs new event kinds behind one prop.

**Sharpest security risk I am introducing (correctly framed):** a richer
**configuration** surface — theming vars, `style`/`className`/`data-*`
passthrough, `messages`, controlled `open`, `rng` — is where a **T2 attacker who
controls the rendering surface can manufacture a false "same."** The events
themselves are near-harmless (values are public; observation doesn't distort).
The danger is any **knob** that can (a) recolor a `≠` verdict into verdict-green,
(b) erase or desync the walk focus ring, (c) transform/overlay one figure
relative to the other, (d) relabel the verdict via `messages`, (e) make the
attacker-chosen-reference journey headless while suppressing the §2.4 scoping
copy, or (f) inject a predictable `rng`/relay the live check-order so the
attacker pre-forges exactly the sampled cells. My mitigations pin every one of
these to the closed-profile boundary and the verdict machine's authority (§4, §5).

## 2. Event catalog

### 2.1 The `EntvizEvent` union and the `onEvent` contract

Every component gains an optional `onEvent?: (e: EntvizEvent) => void`. It is a
**notify-only firehose**: the return value is ignored (except the single
advisory-cancelable `fetch.start`, §4), it never blocks rendering, and it fires
*in addition to* the retained specific callbacks. Each event has a discriminant
`type`, a monotonic `seq`, a `ts` (ms epoch), and a `source`. Payloads are
**object-shaped** so fields can be added without breaking callers.

Because value confidentiality is out of scope, **payloads may carry content**
(value/reference bytes) — they are public. The only content that is
*withheld-by-default* is behind `includeContent` (§3), a light convenience gate
for the corner-case pasted secret, **not** a security firewall. The **sensitivity
tag** is retained purely as a host-side **routing hint** (does this event touch
the network / carry content), not as a defense.

```ts
type EntvizSensitivity = "plain" | "network" | "content"; // ROUTING HINT, not a firewall
type EntvizSource = "entviz" | "pill" | "compare" | "walk" | "voice";

interface EntvizEventBase {
  seq: number;                 // monotonic within a component instance
  ts: number;                  // Date.now() at emit
  source: EntvizSource;
  sensitivity: EntvizSensitivity;
}

type EntvizEvent =
  // ---- lifecycle (Entviz / Pill) ----
  | (EntvizEventBase & { type: "render.error"; message: string })
  | (EntvizEventBase & { type: "disclosure.change"; state: DisclosureState; prev: DisclosureState })
  | (EntvizEventBase & { type: "copy"; kind: CopyKind; ok: boolean })
  | (EntvizEventBase & { type: "display.resize"; fontSizePt: number })
  | (EntvizEventBase & { type: "display.reshape"; targetAr: number; cols: number; rows: number })
  | (EntvizEventBase & { type: "display.tab"; tab: "reference" | "voice" })
  // ---- acquisition (Compare) ----
  | (EntvizEventBase & { type: "reference.acquired"; provenance: Provenance;
                          medium: Medium | null; byteLength: number; origin?: string; content?: string })
  | (EntvizEventBase & { type: "reference.cleared" })
  | (EntvizEventBase & { type: "reference.mediumDetected"; medium: Medium; isUrl: boolean })
  | (EntvizEventBase & { type: "reference.readError"; reason: string })
  | (EntvizEventBase & { type: "secret.detected"; where: "value" | "reference" | "both" })
  // ---- fetch (Compare, network) ----
  | (EntvizEventBase & { type: "fetch.start"; origin: string; url?: string; preventDefault: () => void })
  | (EntvizEventBase & { type: "fetch.success"; origin: string; status: number; byteLength: number; durationMs: number })
  | (EntvizEventBase & { type: "fetch.error"; origin: string; message: string })
  // ---- outcome (Compare) — the load-bearing event ----
  | (EntvizEventBase & { type: "verdict.change"; verdict: VerdictState;
                          medium: string | null; provenance: Provenance | null; coverageBits?: number; complete?: boolean })
  // ---- verification (Walk / Voice, forwarded up through Compare) ----
  | (EntvizEventBase & { type: "walk.start"; mode: "spot-check" | "complete" })
  | (EntvizEventBase & { type: "walk.step"; feature: string | null; index: number })   // suppressed inside a live ceremony (§5)
  | (EntvizEventBase & { type: "walk.complete"; status: WalkStatus })
  | (EntvizEventBase & { type: "voice.start"; mode: "voice-only" | "paste-bind" })
  | (EntvizEventBase & { type: "voice.complete"; status: CeremonyStatus });
  // NOTE: there is deliberately NO `voice.step` — the live check-order never leaves the endpoint (§5).

// VerdictState = "pending" | "different" | "no-difference" | "identical" | "unknown"
```

`onEvent` may throw in host code; components MUST wrap the call in try/catch so a
host bug can't wedge the widget into a state that skips a safety.

**Why `walk.step` carries feature *kind* not glyphs:** partly because the glyph
is unhelpful telemetry, but the *load-bearing* reason (integrity, not
confidentiality) is that the **ordered sequence of steps is the CSPRNG-selected
check order** (§14.2). Even the address/kind sequence, relayed to a counterparty,
is a steering aid. Hence the hard rule: **no `*.step` event fires while a live
ceremony is active, and `voice.step` does not exist at all** (§5).

**⚑ Theming (`--entviz-*` CSS custom properties) — the sharpest T2 surface, new

`--entviz-compare-action` (verdict-adjacent action color), plus many
`--entviz-pill-*`. A **T2 attacker who controls surrounding CSS** (a hostile
embedding page, ancestor stylesheet, or a compromised theme) can set these on any
ancestor and the component inherits them. This is not a prop the host passes — it
is ambient CSS — so the guard cannot be "document it." It must be **structural**:
the **verdict symbol/tone that distinguishes `=` from `≠`, and the walk focus
ring/scrim, must not be defeatable by an inherited CSS variable.** See §5 items
1–3. The remaining, non-verdict `--entviz-*` vars (menu backgrounds, radii, pill
chrome) stay freely themeable — they can't manufacture a false-same.

all five. Legacy scalar callbacks (`onResize(fontSizePt)`, `onVerdict(v)`, …)
stay intact for source-compat.

## 4. Payload & safety policy (integrity-framed)

**1. Content in payloads is acceptable; the value is public.** Payloads may carry
the value, reference, comparison text, and fetched body. The metadata-vs-content
firewall of v1 is **removed** — it defended an out-of-scope asset. The one
convenience gate, `includeContent` (off by default, Compare-only, pill-forbidden),
exists solely so a pasted **secret** (rare corner case) isn't ambiently streamed
to a host logger — a courtesy, not a security boundary. Sensitivity tags are a
host-side **routing hint**, not a defense.

**2. The verdict machine stays authoritative and untamperable.**
- `verdict.change` is **notify-only** — no host code can veto, delay, upgrade, or
  synthesize a verdict. `=`/green is reserved for machine-IDENTICAL
  (`EntvizCompare.ts:173`; comparison-design §3).
- The verdict **symbol** (`=`/`≠`/`?`/`≈`/`…`, hard-coded in `chipFor`,
  `EntvizCompare.ts:165–180,470`) is the primary discriminator and is **not**
  `messages`-override-able and **not** CSS-var-recolorable into ambiguity (§5.1).
- `messages` may localize surrounding prose ONLY, never the verdict
  symbol/tone/semantics nor the §2.4 scoping copy (§5.2).

**3. The walk/ceremony sampling stays unpredictable and local.**
- `rng` defaults to the platform CSPRNG and is **compiled out of production**
  (§3, §5.4). A prod bundle ignores an injected `rng`.
- **No `*.step` event fires while a live ceremony is active, and `voice.step`
  does not exist.** The live check-order never leaves the endpoint (§15.7
  local-only evidence; §15.9 no-joint-seed). `walk.step` fires only for the
  single-user machine walk, feature-kind + index only (§5.5).

**4. Attacker-chosen-reference is always scoped.**
- Provenance chrome and the §2.4 "equal to THIS reference, not trustworthy"
  copy render **regardless of controlled `open`/`disclosure`** (§5.3).
- `verdict.change` carries `provenance` so a host's own telemetry records that an
  affirmative was against a `provided` (host-supplied) reference, not a
  human-acquired one.

**5. Fetched/injected bytes are attacker-authorable → §6.2 gauntlet.**
- A host `fetchReference` result, and any URL-fetched body, is routed through the
  same **strict closed-profile validation → recompute → re-render through OUR
  pinned font → self-consistency** path as pasted bytes (comparison-design §6.2;
  `classifyResult` → `compareSvg`, `EntvizCompare.ts:85–87`). It **cannot** hand
  back a pre-classified IDENTICAL. `fetch.start` is the one advisory-cancelable
  event (a host may block egress; fail-closed — blocking can only deny, never
  force a pass). No event can *cause* or *upgrade* an affirmative verdict.

**6. The closed profile is the passthrough boundary.** `style`/`className`/`data-*`
reach the **chrome root only** — never the entviz `<svg>` and never the walk
overlay SVG (`EntvizWalk.ts:89–118`). Passthrough is **allowlisted** (§3, §5.6).

## ⚑ 5. Security-sensitive decisions (handoff to the adversary)

*Each is an attackable claim: I propose X; safe because Y; risk if wrong is Z. All
framed against the real assets — a false "same" and SVG integrity — under the
tier that reaches them. The adversary attacks these next.*

**5.1 — I propose the verdict SYMBOL (`=` vs `≠`), not just its color, be the
load-bearing discriminator, and that the verdict tone NOT be defeatable by an
inherited `--entviz-*` CSS variable.**
*Asset:* belief that two values are (not) equal. *Tier:* **T2** (attacker
controls surrounding CSS). *Win:* false "same".
Safe because the symbol is hard-coded (`EntvizCompare.ts:173–174,470`) and I will
require the good/bad tones to be **fixed, non-inheritable** for the verdict chip —
either literal colors or CSS vars scoped so an *ancestor* cannot override them
(e.g. set on the component root with `!important` or a private var namespace the
host can't reach), so a T2 stylesheet cannot set `--entviz-compare-bad: #1a7f37`
(`EntvizCompare.ts:193–198`) and repaint a `≠` in verdict-green.
**Risk if I'm wrong:** if the tone stays an ancestor-overridable var AND a user
relies on color over the symbol (plausible — green/red is the loud channel), a T2
page recolors DIFFERENT to look like IDENTICAL. The adversary should decide
whether the verdict chip must ship *fixed* colors (no theming at all on the
verdict tone) versus a scoped-var scheme, and whether the symbol alone is a
sufficient backstop if color is subverted.

**5.2 — I propose `messages` may override surrounding prose ONLY; the verdict
symbol, tone-word, and the §2.4 scoping copy are removed from the override-able
catalog.**
*Asset:* equality belief. *Tier:* the host itself (T2-adjacent: the host authors
`messages`; a compromised/careless host is the threat). *Win:* judgment-tamper →
false "same".
Safe because today `chipFor` reads `m.identical`/`m.different`/`m.pending`
(`EntvizCompare.ts:173–178`) and `m.recognitionNote` (`EntvizWalk.ts:172`;
`EntvizVoiceCompare.ts:97`) from the override-able catalog — I move the verdict
**semantics** out of `messages` so a host cannot relabel DIFFERENT as "Match" or
delete the "does not vouch for the reference" copy.
**Risk if I'm wrong:** if any verdict-bearing string stays in `Partial<CompareMessages>`
(`EntvizCompare.ts:48,202`), a host (or an attacker who supplies a "translation
pack") swaps the verdict text and the symbol becomes the only defense — and if a
future refactor makes the symbol data-driven too, both fall. The adversary should
enumerate exactly which catalog keys are verdict-bearing and confirm none remain
override-able.

**5.3 — I propose provenance chrome + the §2.4 scoping copy render regardless of
controlled `open`/`disclosure`, and `verdict.change` carries `provenance`.**
*Asset:* equality belief (reference-authenticity). *Tier:* T1/social — attacker
supplies the reference. *Win:* a green verdict against an attacker-chosen
reference read as "trustworthy".
Safe because the existing `reference` prop (`EntvizCompare.ts:39,250`) already
lets a host inject a reference; controlled `open` + `verdict.change` only make the
journey *observable/automatable*, and I keep provenance first-class (§2.4) under
host control.
**Risk if I'm wrong:** controlled disclosure + provided reference makes the whole
attacker-chosen-reference path headless (auto-open → inject ref → read verdict);
if a host CSS/`messages`/layout choice can *also* hide the provenance line or the
scoping copy, the user sees a bare green with no "equal to THIS reference"
framing — the F-8 social-engineering path. The adversary should confirm provenance
+ scoping copy are *structurally* non-suppressible (not just present-by-default).

**5.4 — I propose `rng` injection is COMPILED OUT of production builds (prod
ignores it, always CSPRNG); dev/test may inject.**
*Asset:* equality belief. *Tier:* T1+T6 (attacker who knows the sample order
pre-forges the sampled cells). *Win:* false NO-DIFFERENCE.
Safe because a predictable/seeded `rng` in prod defeats the unpredictable-order
anti-habituation (§14.2) and the ceremony's selection-unpredictability that
substitutes for the removed commitment (§15.3/§15.9); a compile-out means a prod
bundle can't honor a seeded `rng` even via config drift or a compromised host.
**Risk if I'm wrong:** if `rng` is merely runtime-warned (v1/Open-Q3), a seeded
source shipped for a Storybook/demo and left on — or a host-controllable `rng`
with a compromised host — lets the attacker prepare exactly the cells the walk
will sample; every non-sampled cell (where the substitution hides) is never
looked at. Silent false-NO-DIFFERENCE. The adversary should decide whether the
compile-out is airtight (tree-shaking, `NODE_ENV`) and whether Walk/Voice/Compare
all honor it uniformly (Voice already takes `rng`, `EntvizVoiceCompare.ts:62`;
Walk hard-codes `csprng`, `:216` — the new prop widens the surface).

**5.5 — I propose NO `*.step` event fires during a live ceremony, `voice.step`
does not exist, and `walk.step` (single-user only) carries feature-kind + index.**
*Asset:* equality belief. *Tier:* T2/remote-steering peer. *Win:* a peer who
learns the next cell pre-positions → false "same".
Safe because the ordered step sequence IS the CSPRNG/authenticator selection
(§14.2/§15.3); §15.7 mandates local-only evidence and §15.9 removes the steering
channel "by construction (no joint seed)". Suppressing the live order keeps it
local.
**Risk if I'm wrong:** the live cell address is computed and rendered on-screen
already (`EntvizVoiceCompare.ts:141,193`); if a host wires `onStep`→a data channel
to the counterparty (which the *existing* `EntvizVoiceCompare.onStep` prop,
`:58`, already permits!), the peer sees the next address before the reader speaks.
**This is a live gap in the current surface, not just the new firehose** — the
adversary should decide whether `EntvizVoiceCompare.onStep` itself must be
constrained (e.g. fire only for local ring-drawing, never carry the address to a
relayable form), and add a hard test: no step address leaves the endpoint during a
ceremony.

**5.6 — I propose `style`/`className`/`data-*` passthrough reaches the chrome root
ONLY (never the `<svg>` or the walk overlay), via an allowlist.**
*Asset:* SVG integrity (secondary) + equality belief (a distorted/overlaid figure
→ false "same"). *Tier:* T2. *Win:* silent alteration / distortion / desync.
Safe because `Entviz.ts:41–51` holds only while the SVG carries renderer-produced
attributes exclusively; an allowlist (`^(data-[a-z0-9-]+|aria-[a-z]+|id|role|title)$`,
string values) forbids `on*`/`href`/`style`/`transform` landing on or near the
glyph.
**Risk if I'm wrong:** a blocklist ("exclude `on*`/`href`/`style`") fails open on
the next dangerous attribute (`xlink:href`, `formaction`, `srcdoc`, data-URI
`style`, `is=`, `transform`, `filter`); a `transform`/`filter`/`mix-blend-mode`
that reaches a figure panel distorts one figure relative to the other (desync →
false-same) even without any script injection. The adversary should confirm the
allowlist is closed and that host `style` on the Compare root cannot cascade a
`transform`/`filter` onto a `figureBox`.

**5.7 — I propose the walk focus ring/scrim not be erasable by inherited
`--entviz-walk-*` CSS vars, and the overlay be host-transform-proof.**
*Asset:* equality belief (the walk directs attention to the checked feature).
*Tier:* T2. *Win:* defeat the walk → user "checks" nothing → false "same".
Safe because §14.3 promises the ring is drawn in the **tool-controlled container**
at fixed scale, "host-transform-proof"; I will pin the ring/scrim colors so an
ancestor can't set `--entviz-walk-ring: transparent` + `--entviz-walk-scrim:
transparent` (`EntvizWalk.ts:73–74`) to erase the spotlight, and ensure the
overlay geometry (`overlayStyle`, `preserveAspectRatio:"none"`,
`EntvizWalk.ts:99,409`) survives a host `transform` on the figure box.
**Risk if I'm wrong:** if the ring color stays an ancestor-overridable var, a T2
page makes the ring/scrim invisible; the walk still asks "do the highlighted
characters match?" but nothing is highlighted, so the user answers "same" on an
unexamined cell. The adversary should decide whether the ring/scrim must ship
fixed colors and whether the overlay is provably aligned when the host applies a
`transform`/`zoom` to an ancestor.

**5.8 — I propose a host `fetchReference` supplies bytes that MUST run the §6.2
gauntlet before reaching a verdict or the SVG engine.**
*Asset:* equality belief + SVG integrity. *Tier:* T1 (attacker-authored SVG
bytes). *Win:* an SVG whose `<text>` says X while its ink shows Y reaches
IDENTICAL with no preimage.
Safe because §6.2 requires strict closed-profile validation, recompute from the
recovered core, re-render through OUR pinned font, and self-consistency before
IDENTICAL; the injected-fetcher path routes through the same `compareSvg`
(`EntvizCompare.ts:85–87`) as pasted bytes.
**Risk if I'm wrong:** if the injected-fetcher result is trusted *because the host
fetched it* and skips any step of the gauntlet, a host fetcher returning a
near-conformant polyglot SVG can forge IDENTICAL. The adversary should require a
test: injected-fetcher bytes get *identical* §6.2 treatment to pasted bytes.
(SSRF/egress from the fetcher is a host-app concern outside entviz's asset frame;
I do not guard it here — flagged so the adversary doesn't re-litigate it as an
entviz asset.)

**5.9 — I propose `verdict.change` (and all verdict/verification events) are
notify-only; only `fetch.start` is advisory-cancelable, enforced in the type
system.**
*Asset:* equality belief. *Tier:* the host. *Win:* a host forces/suppresses a
verdict.
Safe because only `fetch.start` carries `preventDefault` in the union (§2.1), and
cancelling a fetch is fail-closed (deny egress only). No verdict/complete/secret
event is cancelable.
**Risk if I'm wrong:** if `preventDefault` semantics leak to a `verdict.change`
or `*.complete` variant by a future careless addition, a host could veto a
DIFFERENT. The adversary should require a compile-time test that only
`fetch.start` has `preventDefault`, and a test that `walk.complete`/`voice.complete`
cannot flip a `different`/`inconclusive` to `no-difference`.

**5.10 — I propose `allow` is restrict-only and allowlist-closed (present-but-partial
= unlisted keys OFF).**
*Asset:* (mostly a host-policy surface, low asset-relevance). *Tier:* n/a to the
primary asset. *Win:* none directly — a correctness/least-surface claim.
Safe because a *present* `allow` object treats unlisted keys as OFF and only an
*absent* prop is all-on; `allow.url ?? true` is banned.
**Risk if I'm wrong:** a naive `?? true` reducer leaves url/fetch live when a host
passes `allow={{paste:true}}`, so a host that thinks it disabled fetch hasn't.
This is a least-surface/DLP nicety, not a false-same vector — I flag it as lower
priority than 5.1–5.7, correcting v1's over-weighting of it.

**5.11 — I propose the shared size/shape (`dispFs`/`dispAr`) drives BOTH figures,
and neither `display.resize`/`display.reshape` nor host CSS may desync them.**
*Asset:* equality belief. *Tier:* T2. *Win:* two figures at different size/shape
read as "same" (or a mismatch hidden by scale) → false verdict-by-eye.
Safe because both panels render from one `dispFs`/`dispAr` state
(`EntvizCompare.ts:225–227,401,451`) so the machine keeps them synced.
**Risk if I'm wrong:** a host CSS `transform: scaleX()`/`zoom` on one panel (via
`style` cascade, 5.6) resizes one figure independently of the other, so the human
walk/voice comparison is between mismatched renderings. The adversary should
confirm the figure boxes are isolated from host `style` and that no prop exposes
per-figure size.

## 6. Open questions (each with a recommended default)

1. **Verdict tone: fixed colors, or a private-namespace CSS var an ancestor can't
   reach?** *Recommended:* ship the verdict good/bad tones as **fixed colors** (no
   theming on the verdict chip's semantic color), keep the symbol as the primary
   discriminator, and let only *non-verdict* chrome (menus, radii, pill body) stay
   `--entviz-*`-themeable. Revisit if a host demands verdict-color theming — but
   the default must be un-subvertible-by-ancestor-CSS (5.1).

2. **Should the walk ring/scrim colors be themeable at all?** *Recommended:* **no**
   — pin `RING`/`SCRIM` (`EntvizWalk.ts:73–74`) to fixed values; the spotlight is a
   security affordance, not decoration (5.7). A host that needs contrast tuning can
   file an issue; ambient-CSS erasure is the threat.

3. **Which `messages`/`CompareMessages` keys are verdict-bearing and must leave the
   override-able catalog?** *Recommended:* move `identical`/`different`/`pending`/
   `unknownReason`/`recognitionNote` (verdict labels + scoping copy) into a
   **non-override-able** internal catalog; keep acquisition/UI prose
   (`pastePrompt`, `dropHint`, `fetchHint`, tab labels) override-able (5.2). Confirm
   the exact split against `compare-messages.ts`.

4. **Must `rng` be compiled out, or is a dev-only prop + prod warning enough?**
   *Recommended:* **compile out in prod** (`NODE_ENV`/build-flag gate) so a prod
   bundle always uses CSPRNG; warning is additional (5.4). This is stricter than
   v1's Open-Q3 and is the right call under the sampling-integrity frame.

5. **Should `EntvizVoiceCompare.onStep` (existing) be constrained so the live cell
   address can't be relayed to the counterparty?** *Recommended:* yes — document
   `onStep` as **local-ring-drawing only, never relay to a remote party**, and
   consider firing it with a non-address token (an opaque local index) so a host
   physically cannot forward the address (5.5). This is a pre-existing surface, not
   introduced by the firehose.

6. **Does `includeContent` need to exist at all, given confidentiality is out of
   scope?** *Recommended:* keep it as a **light convenience default-off** flag
   purely for the pasted-secret corner case; drop v1's `unsafe*` naming, the
   imperative-pull debate, the byteLength-oracle bucketing, and the maxSensitivity
   cap — all defended a non-asset. If the adversary prefers, content-in-events can
   even be **on by default** (value is public) with the flag only *suppressing*
   secret-shaped content; that is a DevX call, not a security one.

7. **Payload identifiers vs. display labels for `disclosure.change`.**
   *Recommended:* payload uses stable internal identifiers
   (`"pill"|"visualize"|"compare"`); display labels (Cite/Reference/Look/Compare)
   stay localizable chrome and never enter the event contract, so a label rename
   can't break a host's analytics — and can't tamper the verdict semantics (ties to
   5.2).
